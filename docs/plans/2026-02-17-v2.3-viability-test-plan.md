## What Changed Since V2.2

One targeted fix:

| Change | Impact |
|--------|--------|
| Multi-signal entity ranking | When no `context_file` is provided, `resolve_entity()` now scores candidates by: (1) language mismatch penalty, (2) generated code penalty (`generated/`, `node_modules/`, `.d.ts`), (3) edge count (more relationships = better), (4) alphabetical tiebreaker. Previously: alphabetical only. |
| `--language` flag on trace | CLI `--language python` and MCP `language="python"` parameter deprioritize entities from other languages. |

**Expected impact:** B2 was the most winnable flip. In V2.2, `clew trace "PrescriptionFill"` resolved to TypeScript generated type (alphabetically first) instead of the Python Django model. With the fix, the Python model wins: it has more edges AND the TypeScript path contains `generated/`. The `--language` flag provides an additional override if needed.

## Methodology Improvements Over V2.2

V2.2's meta-evaluation (methodology critic rated 2.5/5.0) identified these weaknesses. V2.3 addresses each:

### 1. True Transcript Sanitization (was: tool names visible)

**V2.2 problem:** Transcripts were labeled Alpha/Beta, but tool names (`clew search`, `Grep(...)`) and output formats (JSON with `score` fields vs raw file content) were visible. Scorers could identify tools.

**V2.3 fix: Dedicated sanitization pipeline**

After exploration, a **sanitization agent** (separate from explorers and scorers) processes each raw transcript:

| Original | Sanitized |
|----------|-----------|
| `clew search "query" --json` | `search("query")` |
| `clew trace "entity" --json` | `trace("entity")` |
| `Grep(pattern="...", ...)` | `search("...")` |
| `Glob(pattern="...")` | `find_files("...")` |
| `Read(file_path="...")` | `read("...")` |
| Score fields (`"score": 0.94`) | Removed |
| Qdrant metadata (`chunk_type`, `enriched`) | Removed |
| Line count metadata (`line_start`, `line_end`) | Kept (both tools produce this) |
| File paths | Kept (both tools produce this) |

Output content (code snippets, file contents) is kept verbatim — only tool invocation syntax and tool-specific metadata are normalized.

**Verification gate:** After sanitization, a separate **verification agent** reads each sanitized transcript and attempts to guess which tool was used. If it guesses correctly with >60% confidence on >2 of 8 tests, the sanitization is insufficient and must be revised before scoring proceeds.

### 2. Multiple Scorers Per Test (was: 1 scorer, arithmetic error)

**V2.2 problem:** 1 scorer per test. The A1 scorer made an arithmetic error (calculated 4.25 vs 4.25, actual was 4.35 vs 4.15).

**V2.3 fix:** 2 independent scorers per test (16 total). Each scorer provides raw dimension ratings only (integers 1-5). Weighted averages are computed programmatically — scorers never calculate aggregates. If scorers disagree by >1 point on any dimension, a third tiebreaker scorer is invoked.

### 3. Pre-Registered Methodology (was: post-hoc corrections)

**V2.2 problem:** The A1 correction and relational insight debate were resolved after results were in.

**V2.3 fix:** This plan IS the pre-registration. All rules are frozen before any agent runs. Specifically:
- Weighted average formula is fixed (30/25/20/15/10)
- Winner determination is mechanical (higher weighted average wins; ties go to higher completeness score)
- Relational insight is scored for BOTH tools (V2.2 showed grep agents DO surface connections via manual code reading — forcing N/A was unfair to grep and inaccurate)
- No post-hoc corrections to scores. If a scorer makes an error, the mechanical formula overrides their stated conclusion.

### 4. Controlled Grep Access (was: clew agents had full bash)

**V2.2 problem:** Clew agents could (and did) fall back to grep via Bash. The comparison was "clew+grep vs grep-only."

**V2.3 fix:** Keep the same setup. This is the realistic comparison — a clew user still has grep available. But transcript sanitization makes this less of a concern for blind scoring, since both agents' search calls are normalized to the same `search("...")` format.

## Scoring Rubric

### Dimensions (unchanged from V2.2)

| Dimension | Weight | What It Measures |
|-----------|--------|------------------|
| Discovery Efficiency | 30% | How quickly the agent reaches actionable understanding |
| Context Precision | 25% | Signal-to-noise ratio in loaded context |
| Completeness | 20% | Coverage of key artifacts per test checklist |
| Relational Insight | 15% | Connections surfaced that the agent did not explicitly search for |
| Answer Confidence | 10% | Whether the final output is actionable without further investigation |

### Rating Scale (unchanged)

Each dimension rated 1-5 by each scorer. The weighted average is computed programmatically:

```
weighted_score = (discovery * 0.30) + (precision * 0.25) + (completeness * 0.20) + (relational * 0.15) + (confidence * 0.10)
```

### Winner Determination (new: mechanical, pre-registered)

```
if clew_weighted > grep_weighted:
    winner = "clew"
elif grep_weighted > clew_weighted:
    winner = "grep"
else:  # exact tie
    winner = "higher completeness score"  # tiebreaker
```

When 2 scorers disagree on winner direction, use the average of both scorers' weighted scores.

### Relational Insight Clarification (new)

Both tools are scored on relational insight. V2.2 planned N/A for grep but scorers awarded scores anyway (and correctly — grep agents DO discover connections through manual code reading). Scoring both tools on all 5 dimensions is fairer and simpler.

## Viability Thresholds (unchanged from V2.2)

| Verdict | Criteria |
|---------|----------|
| **Ship** | Clew avg >= 3.50 AND wins >= 5/8 |
| **Iterate** | Clew avg >= 2.50 AND wins >= 3/8 |
| **Kill** | Below Iterate thresholds |

### What Ship means

Entity resolution fix + existing strengths are sufficient. Move to next version (exhaustive mode, peripheral code surfacing, etc.) as feature work, not viability-blocking.

### What Iterate means

Entity resolution fix helped but wasn't enough. Likely need exhaustive mode or another structural improvement before shipping.

## Execution Pipeline

### Phase 0: Pre-flight (5 min)

- [ ] Qdrant running: `docker compose up -d qdrant`
- [ ] Entity resolution fix installed: `pip3 install -e .`
- [ ] Re-index target codebase: `clew index /Users/albertgwo/Work/evvy --full`
- [ ] Verify fix: `clew trace "PrescriptionFill" --json` resolves to Python model (not TypeScript)
- [ ] Verify compact mode: `clew search "test query" --json` returns snippet-based results
- [ ] Verify trace: `clew trace "_process_shopify_order_impl" --direction outbound --depth 1 --json` returns edges

### Phase 1: Exploration (16 agents, parallelizable per test)

For each of the 8 tests, launch 2 exploration agents:

**Agent A (grep-only):**
- Tools: Grep, Glob, Read (in target codebase)
- No Bash, no clew access
- Receives only the scenario description
- No access to test plan, CLAUDE.md, or codebase documentation
- Working directory: target codebase root

**Agent B (clew):**
- Tools: Bash (for clew CLI), Read (for follow-up)
- Receives only the scenario description
- No access to test plan, CLAUDE.md, or codebase documentation
- Working directory: target codebase root
- Agent does NOT receive hints about `--language` flag — must discover it organically via `--help` (or not)

Each agent produces a transcript (tool calls + reasoning + final answer).

**Parallelization:** All 8 test pairs can run in parallel (16 agents total). Each agent is independent.

### Phase 2: Sanitization (8 agents, 1 per test)

For each test, a sanitization agent:
1. Reads both raw transcripts
2. Randomly assigns Alpha/Beta labels (coin flip, recorded in mapping file)
3. Applies the tool-name and metadata normalization table
4. Produces two sanitized transcripts per test
5. Records the Alpha/Beta → clew/grep mapping in a sealed file

Output: 16 sanitized transcripts + 1 mapping file.

### Phase 3: Verification (1 agent)

A single verification agent reads all 16 sanitized transcripts and attempts to identify which tool each agent used. Reports confidence per test. If >60% confidence on >2 tests, Phase 2 must be re-run with stricter sanitization.

### Phase 4: Scoring (16 agents, 2 per test)

For each test, 2 independent scoring agents:
1. Receive both sanitized transcripts (Alpha and Beta) for that test
2. Receive the scoring rubric and dimension definitions
3. Rate each transcript on 5 dimensions (integer 1-5)
4. Do NOT compute weighted averages (done programmatically)
5. Do NOT declare a winner (done programmatically)

**Scorer instructions (exact prompt):**

> You are scoring two agent transcripts that attempted the same codebase exploration task. Score each agent independently on 5 dimensions using the rubric provided. Provide integer ratings (1-5) for each dimension. Do NOT compute aggregates or declare a winner — just rate each dimension.
>
> IMPORTANT: Score each agent against the absolute rubric, not relative to the other agent. A score of 3 means "good" regardless of what the other agent scored.

### Phase 5: Computation (programmatic, no agents)

A script:
1. Reads all scorer ratings
2. Computes weighted averages per test per agent
3. Determines winners per test
4. Unmasks Alpha/Beta using the mapping file
5. Computes category and overall averages
6. Applies viability thresholds
7. Flags any scorer disagreements (>1 point on any dimension)
8. Outputs the final results table

### Phase 6: Disagreement Resolution (conditional)

If any scorer pairs disagree by >1 point on any dimension:
1. Launch a third tiebreaker scorer for that test
2. Use the median of 3 scores for each dimension
3. Re-run Phase 5 computation

## Expected Outcomes

### Optimistic (Ship)

Entity resolution fix flips B2. Clew's semantic search + correct trace produces a competitive or winning transcript. Combined with V2.2's existing 4 wins (A1, A3, A4, B1), this yields 5-3 wins. Average stays above 3.50 because B2 was clew's worst score (2.95) and improving it lifts the average.

**Predicted: Ship** (avg ~4.0, wins 5/3)

### Realistic (Strong Iterate → borderline Ship)

Entity resolution fix improves B2 from 2.95 to ~3.5-4.0 but doesn't produce a decisive win (grep's exhaustive model enumeration still scores higher on completeness). Meanwhile, natural variance means 1-2 other tests shift slightly. Final result: 4-4 or 5-3 with average around 3.50.

**Predicted: borderline Ship/Iterate** (depends on individual test variance)

### Pessimistic (Iterate)

The exploration agent doesn't trigger `PrescriptionFill` trace in B2 (different exploration path), or the sanitization verification fails repeatedly, or scorer disagreements produce noisy results. Entity resolution fix helps technically but doesn't manifest in evaluation.

**Predicted: Iterate** (similar to V2.2)
