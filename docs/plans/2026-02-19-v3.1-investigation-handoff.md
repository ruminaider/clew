# V3.1 Investigation Handoff

**Goal:** Diagnose why V3.0 features were invisible to evaluation agents, fix the visibility gaps, and determine if V3.0's features actually improve agent performance when agents can see them.

**Context:** V3.0 scored KILL (3.84/5.0, 3/8 Track A wins vs V2.3's 3.96/5.0, 5/8 wins). The root cause is NOT that V3.0 features don't work — it's that evaluation agents never saw them. Post-hoc transcript analysis found zero evidence of confidence labels, ENUMERATION auto-classification, related_files, or grep_results in any Track A transcript.

## Phase 1: Root Cause Diagnosis (Research Only — No Code Changes)

The investigation must answer three questions before any code changes are made.

### Question 1: Per-transcript output mode audit

Read the 8 Track A clew raw transcripts (`.clew-eval/v3.0/raw-transcripts/{A1,A2,A3,A4,B1,B2,C1,C2}-clew.md`) and the 4 Track B clew transcripts (`D1-D4-clew.md`). For each transcript, produce a row in the audit table below.

**Important context:** Transcripts are structured summaries — they show CLI commands but not raw JSON responses. This means we can determine what output mode *was used* (from command flags) but must *infer* whether the agent noticed or acted on V3 metadata fields (from subsequent agent behavior).

For each transcript, extract:

1. **Exact CLI invocations** — Were they `clew search "..." --json`, `clew search "..." --raw`, bare `clew search "..."` (rich text), or MCP tool calls?

5. **`--project-root` used?** — If missing, note the working directory assumption and look for error messages or empty results that could indicate cache resolution failure.

**Known data points from spot-checking:**
- A2 (Track A): Used `--json` with `--project-root` on every call
- D1 (Track B): Used bare `clew search` (rich text, no `--json` or `--raw`)

This reveals a three-way split across output modes, not a binary raw-vs-json question.

The three CLI output modes and their V3 field coverage:
- **`--raw` mode** (`cli.py:203-235`): Outputs only per-result fields (file_path, content, score, chunk_type, line_start/end, language, class/function_name). **NO V3 metadata at all.**
- **`--json` mode** (`cli.py:237-274`): Includes top-level `confidence`, `confidence_label`, `intent`, `suggestion`, plus all result fields. V3 features ARE present (but `mode`, `suggested_patterns`, `related_files`, `grep_total` are still missing — see Phase 2 gap table).
- **Rich output** (default, no flags): Only `intent` is partially shown in a dim header line. All other V3 metadata invisible.
- **MCP responses** (`mcp_server.py:316-346`): Include all V3 fields including `related_files`, `grep_results`, `suggested_patterns`.

This converts the investigation from "which mode?" to "which fields were available AND did the agent use them?" — the answer determines whether the fix is output format (fields missing) or field prominence (fields present but ignored).

## Phase 3: Fix Visibility Gaps

Based on Phase 1-2 findings, implement fixes. Priority order:

### Fix 1: Add V3 metadata to `--raw` output (CRITICAL if agents used `--raw`)

Currently `--raw` outputs only result arrays. Add a metadata wrapper:

```python
# cli.py, raw output path
if raw:
    output = {
        "results": [...],
        "confidence": response.confidence,
        "confidence_label": response.confidence_label,
        "intent": response.intent.value,
        "mode": effective_mode or "semantic",
    }
    if response.suggestion_type.value != "none":
        output["suggestion"] = suggestion_map.get(...)
    if grep_results:
        output["grep_results"] = [...]
```

**Risk:** This changes `--raw` output structure from array to object. Any existing integrations using `--raw` would break. Consider: either change `--raw` (breaking), or add a `--raw-v3` flag, or deprecate `--raw` in favor of `--json`.

**Recommendation:** Since `--json` already has the right structure, the better fix is to ensure evaluation agents use `--json` and to deprecate `--raw` eventually. But for V3.1, keep `--raw` as-is (backwards compatibility) and ensure all evaluation agents use `--json`.

### Fix 2: Add missing fields to `--json` output

**Step 2a: Extract `_surface_peripherals` to shared module**

The `_surface_peripherals` function (and its helper `_categorize_relationship`) currently lives in `mcp_server.py:159-223`. It has zero MCP imports — it only uses `CacheDB.traverse_relationships_batch()` and `SearchResult` attributes. Importing it from `mcp_server.py` into `cli.py` would create a backwards CLI → MCP server dependency.

Extract to `clew/search/surfacing.py` instead:
- Move `_surface_peripherals()` and `_categorize_relationship()` from `mcp_server.py`
- Update `mcp_server.py` to `from clew.search.surfacing import _surface_peripherals`
- Move `tests/unit/test_mcp_server.py::TestPeripheralSurfacing` → `tests/unit/test_search_surfacing.py`

This fits the search post-processing pattern (`clew/search/` already has `rerank.py`, `enhance.py`). Estimated ~10 lines of mechanical refactoring.

**Step 2b: Add fields to `--json` output path in `cli.py`**

```python
# After line 258 in current code
output["mode"] = effective_mode or "semantic"
if response.suggested_patterns:
    output["suggested_patterns"] = response.suggested_patterns

# Add related_files surfacing (now from shared module)

```

## Phase 5: Focused Re-Evaluation Design

After visibility fixes are verified, design a focused 4-test re-evaluation. Do NOT repeat the full 12-test, 72-agent pipeline. Use a lean methodology.

### Test Selection (4 tests)

| Test | Category | Why Selected |
|------|----------|-------------|
| C1 | ENUMERATION | V3.0's weakest category. Tests ENUMERATION auto-classification + keyword mode guidance |
| B1 | Structural | Tests `related_files` peripheral surfacing (should surface test files, callers) |
| D1 | Exhaustive | Tests exhaustive mode + grep integration visibility |
| D4 | Exhaustive | Tests mode selection guidance (agent should use exhaustive for enumeration) |

### Methodology

- Same blind A/B framework as V3.0 (sanitize, verify, score)
- 2 agents per test (8 exploration agents total)
- 2 scorers per test (8 scoring agents)
- Same rubric (5 dimensions, same weights)
- 1 verification agent
- **Total: ~17 agents** (vs 72 for V3.0)

### Success Criteria

| Criterion | Threshold |
|-----------|-----------|
| Feature visibility | All 4 V3 features (confidence_label, intent, related_files, suggestion) visible in >=3/4 clew transcripts |
| Agent behavior change | At least 2/4 clew agents adjust behavior based on V3 metadata (e.g., switch modes after low confidence) |
| Win rate | Clew wins >=3/4 tests |
| No regression | Clew avg >=3.50 (above Iterate threshold) |

### Verdict Rules

| Verdict | Criteria |
|---------|----------|
| Ship V3.1 | Wins >=3/4 AND features visible in >=3/4 AND avg >=3.50 |
| Iterate | Wins >=2/4 AND features visible in >=3/4 |
| Kill V3 entirely | Features visible but wins <=1/4, OR features still invisible |

## Phase 6: Decision Point

Based on the focused evaluation, one of three outcomes:

1. **Ship V3.1**: Features are visible, agents use them, performance improves. Proceed to full regression evaluation (repeat V2.3's 8 tests) before final ship.

2. **Iterate to V3.2**: Features are visible but agents don't consistently use them. Need stronger guidance mechanisms (e.g., auto-suggesting mode in tool description, prepending suggestion to results).

3. **Kill V3 line**: Even with visibility fixed, V3 features don't improve performance. Accept that semantic search modes don't help agents and focus V4 on a different approach (e.g., deeper trace integration, better enrichment, or embedding quality improvements).

## Anti-Patterns to Avoid

1. **Do NOT add more features.** V3.0 already has 5 features. The problem is visibility, not capability.
2. **Do NOT redesign the search pipeline.** Confidence scoring, ENUMERATION routing, and grep integration all work in the backend.
3. **Do NOT expand the evaluation.** 4 focused tests are sufficient to answer "do visible features help agents?" A 12-test rerun wastes resources.
4. **Do NOT ship V3.1 without confirming visibility first.** The entire V3.0 KILL was caused by invisible features. Shipping more invisible features would repeat the same failure.
