## Why This Plan Exists

The previous viability tests (V1: 1.72, V2: 2.17, V2.1: 1.36) measured the wrong thing. They asked: "Can clew find function X faster than `Grep("def X")`?" This is the wrong question because:

1. **The baseline was rigged.** The baseline agent was handed the exact grep query (e.g., `Grep("def _process_shopify_order_impl")`). A real agent doesn't know the function name -- it starts with a vague problem description and has to discover what to search for. The previous tests measured lookup speed, not discovery efficiency.

2. **Raw JSON mode inflated response sizes.** V2.1 tested raw `clew search --raw` output (7K-181K chars per call). The intended user experience is compact MCP responses (~20x smaller). Testing raw mode measured an artifact of the CLI, not the tool's value to an agent.

3. **Grep is unbeatable at grep's job.** Pattern matching (`find all raise ValidationError`) is fundamentally a grep task. Testing clew against grep on exact-string lookup is like testing a search engine against Ctrl+F -- the search engine will always be slower for known strings, but that's not what it's for.

The revised plan measures what actually matters: **Can clew help an agent reach understanding faster when starting from a vague question?**

---

## Design Principles

### 1. Compact mode by default

All clew commands use the MCP tool interface. Search returns snippets (signature + docstring preview), not full source. The agent can selectively request `detail="full"` for promising results, just as a real agent would. No `--raw` mode.

### 2. Baseline = agent with NO prior knowledge

The baseline agent receives only a scenario description. It must figure out what to grep for, which files to read, and how to piece together understanding. No hints about function names, file paths, or module locations.

### 3. Measure discovery, not lookup

Tests start from scenarios ("A subscription order didn't create a PrescriptionFill"), not identifiers ("find `_process_shopify_order_impl`"). The metric is: how many tool calls and how much context did the agent consume before it could act?

### 4. Keep grep-advantaged tests honest

Category C includes tests where grep is expected to win. These are labeled clearly so the evaluation doesn't pretend clew can replace grep for pattern matching.

### 5. Relational context as a scoring dimension

A new dimension captures whether the tool surfaced connections the agent didn't know to look for -- the unique value proposition of trace and semantic search.

## Test Categories

### Category A: Discovery from Vague Starting Point (clew-advantaged)

These tests start from a scenario or question. Neither agent is given function names, file paths, or any structural hints. They receive ONLY the scenario description below.

---

#### Test A.1: Subscription Renewal Failure Investigation

**Scenario given to both agents:**

> "A customer reported their subscription renewal didn't create a prescription fill. Investigate the code path from order webhook to PrescriptionFill creation and identify where failures can occur."

**What the agent must find to score well:**
- The webhook entry point (Shopify order processing)
- The subscription/refill-specific processing path
- The eligibility/validation logic (can the order be refilled?)
- The PrescriptionFill creation step
- At least one failure mode (validation failure, missing prescription, etc.)

**Why this tests discovery:**
A grep agent must guess search terms. "subscription renewal" might not appear in the code -- the actual terms are "refill," "treatment_refill," "can_refill_entire_order." The agent has to iterate through failed searches to find the right vocabulary. A clew agent's semantic search can bridge the gap between the scenario's language and the code's terminology.

**Clew approach hint (for scorer, not for test agent):**
`search("subscription order creates prescription fill")` -> `trace(entry_point, direction="outbound", max_depth=2)` -> `search("prescription fill creation failure", intent="debug")`

---

#### Test A.2: Adding a New Checkout Source Type

**Scenario given to both agents:**

> "We need to add a new checkout source type for a partner integration. What code needs to change? Provide a comprehensive list of files and code locations that reference or depend on the checkout source concept."

**What the agent must find to score well:**
- The Order model and its `checkout_source` field (or equivalent)
- All places where checkout source is set (order creation paths)
- All places where checkout source is read (conditional logic, reporting)
- Serializers that expose checkout source
- Admin configuration (if checkout source is editable)
- Tests that exercise checkout source behavior

**Why this tests discovery:**
This is an impact analysis task starting from a concept ("checkout source"), not a symbol. The agent must find where the concept lives, then fan out to all its consumers. Clew's trace can automate the fan-out; grep requires the agent to already know what to search for at each step.

---

#### Test A.3: Pharmacy API Error Handling Investigation

**Scenario given to both agents:**

> "The pharmacy API is returning errors for some refill orders. Where in the codebase do we call the pharmacy API, how do we handle errors from it, and what retry/monitoring logic exists?"

**What the agent must find to score well:**
- The pharmacy API client or integration code
- The specific call site(s) for refill submission
- Error handling around pharmacy API calls (try/except, error responses)
- Retry logic (Celery task retry, manual retry, dead letter queue)
- Monitoring or alerting for pharmacy API failures

**Why this tests discovery:**
The agent must trace a multi-hop path: symptom (API errors) -> API call site -> error handling -> retry mechanism -> monitoring. Each hop requires discovering the next piece. Clew's search + trace can surface the full chain; grep requires the agent to follow breadcrumbs one at a time.

---

#### Test A.4: Order Type Determination Logic

**Scenario given to both agents:**

> "Explain how the order processing system decides what type of order it is (e.g., new subscription vs. refill vs. one-time purchase) and how that determination affects downstream processing."

**What the agent must find to score well:**
- The order processing entry point
- The branching logic that determines order type
- At least 2 distinct order type handlers (e.g., new subscription path vs. refill path)
- How order type determination flows into downstream behavior (different fill logic, different notifications, etc.)

**Why this tests discovery:**
This is a "explain the architecture" task. The agent needs to build a mental model, not find a single function. Clew's semantic search can match "order type determination" to relevant code even if those exact words don't appear. Trace can show the branching structure. Grep requires the agent to guess the codebase's vocabulary for order typing.

---

### Category B: Structural Understanding (mixed advantage)

These tests start from a known concept but require understanding structure and relationships.

---

#### Test B.1: Impact Analysis for Signature Change

**Scenario given to both agents:**

> "We're considering changing the signature of the function that processes shopify orders for treatment refills. What would break? Find all callers, understand what arguments they pass, and identify any test coverage."

**What the agent must find to score well:**
- The target function (must discover it from the description, not be told the name)
- All direct callers (inbound dependencies)
- What arguments each caller passes (requires reading the call sites)
- Test coverage for the function and its callers

**Expected advantage:**
- Clew: semantic search finds the function from description; trace finds callers in 1 call
- Grep: agent must guess the function name first, then grep for it; but once found, grep is precise
- Net: clew-advantaged on discovery, comparable on the follow-up reads

---

#### Test B.2: Model Relationship Mapping

**Scenario given to both agents:**

> "Map the relationship between Order, PrescriptionFill, and Prescription models. Include foreign keys, many-to-many relationships, through-tables, and any computed properties that bridge them."

**What the agent must find to score well:**
- Order model and its relevant fields
- PrescriptionFill model and its relevant fields
- Prescription model and its relevant fields
- The M2M through-table (PrescriptionFillPrescriptions)
- Foreign key relationships between them
- Computed properties (e.g., `fill_count`)

**Expected advantage:**
- Clew search: good at finding through-tables and computed properties via semantic match
- Clew trace: weak at model-level relationships (known limitation -- trace operates at function level, not FK level)
- Grep: needs multiple targeted searches but is thorough once the agent knows what models exist
- Net: roughly comparable; clew search helps discovery, grep helps exhaustive enumeration

---

### Category C: Grep-Advantaged Tests (honest assessment)

These tests are included to keep the evaluation fair. They represent tasks where grep's exact-pattern-matching is the right tool. Clew is expected to lose or tie on these.

---

#### Test C.1: Find All Retryable Celery Tasks

**Scenario given to both agents:**

> "Find all Celery tasks in the codebase that have retry configuration (auto-retry, manual retry, max_retries, etc.). List each task with its retry settings."

**What the agent must find to score well:**
- All `@shared_task` or `@app.task` decorated functions with retry config
- Retry parameters: `autoretry_for`, `max_retries`, `retry_backoff`, `self.retry()`
- Complete list (no missed tasks)

**Expected advantage:**
Grep wins. This is a pattern-matching task: search for `@.*task.*retry`, `autoretry_for`, `max_retries`, `self.retry(`. Grep returns exhaustive results. Clew's semantic search may miss tasks whose descriptions don't mention "retry" or may return false positives.

---

#### Test C.2: Count Django Models Inheriting from a Mixin

**Scenario given to both agents:**

> "How many Django models in the codebase inherit from TimestampMixin (or similar timestamp base class)? List them all."

**What the agent must find to score well:**
- The correct mixin name (may not be exactly "TimestampMixin")
- Every model class that inherits from it
- Accurate count

**Expected advantage:**
Grep wins. `Grep("class.*TimestampMixin\|class.*TimeStampedModel", glob="*.py")` returns the complete list in one call. Clew's trace may not resolve inheritance relationships, and semantic search is not designed for exhaustive enumeration.

## Scoring Rubric

### Per-Dimension Definitions

| Dimension | 3 (Excellent) | 2 (Good) | 1 (Weak) | 0 (Failed) |
|-----------|---------------|----------|----------|-------------|
| **Discovery efficiency** | Agent reaches actionable understanding in <=3 tool calls | 4-6 tool calls | 7-10 tool calls | 10+ tool calls or gives up |
| **Context precision** | <=20% of loaded context is noise/irrelevant | 20-40% noise | 40-60% noise | 60%+ noise |
| **Relational insight** | Tool surfaced >=2 connections the agent didn't explicitly search for | 1 unexpected connection | All results were directly requested | N/A (tool provided no relational value) |
| **Completeness** | All key artifacts found (per test checklist) | >=80% of key artifacts found | 50-80% found | <50% found |
| **Answer confidence** | Agent could act on results without additional verification | Needs 1-2 follow-up reads | Needs significant follow-up | Results insufficient to act |

### Dimension Definitions Explained

**Discovery efficiency** measures how quickly the agent gets from "I don't know anything" to "I understand the relevant code." A tool call counts if it returns results the agent uses. Calls that return nothing useful or that the agent ignores still count against the total.

**Context precision** measures signal-to-noise ratio. For grep, this includes false positive matches the agent has to scan through. For clew, this includes irrelevant search results. For Read, this includes lines of code the agent had to load but didn't need. Only context that contributed to the agent's understanding counts as signal.

**Relational insight** captures clew's unique value proposition: surfacing connections the agent didn't know to ask for. This applies when trace returns a caller the agent wouldn't have found via direct search, or when search results include related code the agent didn't know existed. Score N/A if the tool provides no relational features (standard for grep).

**Completeness** is scored against each test's "what the agent must find" checklist. Partial matches count proportionally (e.g., finding 4 of 5 artifacts = 80%).

**Answer confidence** measures whether the agent's final output is actionable. Can someone take the agent's answer and immediately write code / file a fix / plan a change? Or does the answer require more investigation?

## Execution Methodology

### Agent Setup

**Two agents per test, running independently:**

1. **Grep agent:** Has access to Grep, Glob, and Read tools only. Receives only the scenario description. Has no prior knowledge of the codebase structure, file layout, or naming conventions.

2. **Clew agent:** Has access to clew MCP tools (search, trace, explain, get_context, index_status) plus Read (for selective follow-up). Receives only the scenario description. Has no prior knowledge of the codebase structure.

**Critical constraint:** Neither agent reads these test plans, the CLAUDE.md, or any documentation about the codebase. They start cold.

### Clew Configuration

- All search calls use compact mode (default). The agent may request `detail="full"` selectively.
- Default `limit=5` for search and explain.
- Trace uses default parameters unless the agent chooses to narrow (direction, depth, relationship_types).
- Index must be current: `clew status` should show ~13,000 chunks with enrichment.

### Transcript Recording

For each agent run, record:

| Field | Description |
|-------|-------------|
| **Tool calls** | Ordered list of every tool invocation with parameters |
| **Context loaded** | Character count of all tool responses received |
| **Useful context** | Character count of tool response content that contributed to the final answer |
| **Key artifacts found** | Which items from the test's checklist were identified |
| **Unexpected discoveries** | Connections or artifacts the agent found that weren't in its search query |
| **Dead ends** | Tool calls that returned nothing useful |
| **Final answer** | The agent's summary/explanation |

### Independent Scoring

A third agent (the scorer) reviews both transcripts for each test and scores each on the 5 dimensions. The scorer does not know which agent used which tool -- transcripts are anonymized as "Agent Alpha" and "Agent Beta" with tool names redacted (replaced with generic "search_tool", "file_reader", etc.) where possible.

**Scorer instructions:**

> You are scoring two agent transcripts that attempted the same task. Score each agent independently on 5 dimensions (Discovery efficiency, Context precision, Relational insight, Completeness, Answer confidence) using the rubric provided. Do not compare the agents to each other -- score each against the absolute rubric. Note: Relational insight is N/A for agents using only pattern-matching tools.

## Aggregate Scoring

### Per-Test Score

Each test produces two sets of scores (grep agent, clew agent), each with 5 dimensions. The per-test comparison is:

```
Test A.1 Clew advantage = (Clew average - Grep average)
```

For the Relational insight dimension, grep agents always score N/A. When computing averages for grep agents, use 4 dimensions; for clew agents, use all 5. This means clew must earn its relational insight score -- it's not a freebie.

### Category Aggregates

| Category | Tests | Weight | Rationale |
|----------|-------|--------|-----------|
| A: Discovery | A.1-A.4 | 50% | This is the core value proposition |
| B: Structural | B.1-B.2 | 25% | Important but secondary |
| C: Grep-advantaged | C.1-C.2 | 25% | Honesty check |

### Viability Verdicts

| Verdict | Criteria |
|---------|----------|
| **Strong viability** | Clew average >=2.5 on Category A tests AND clew outperforms grep on >=3 of 4 Category A tests AND Category C scores are honestly reported (clew is not inflated) |
| **Conditional viability** | Clew average >=2.0 on Category A AND clew outperforms grep on >=2 of 4 Category A tests |
| **Not viable** | Clew average <2.0 on Category A OR clew fails to outperform grep on any Category A test |

**Important:** "Outperforms" means the clew agent's average score across all 5 dimensions is higher than the grep agent's average across 4 dimensions (since grep has no relational insight). A 0.5+ point advantage is a clear win. A 0.0-0.5 advantage is a marginal win. A negative delta is a loss.

## Expected Outcomes and Interpretation Guide

### What a "Strong viability" result would mean

Clew meaningfully accelerates discovery. An agent with clew reaches understanding in fewer steps, with less noise, and discovers connections it wouldn't have found via grep alone. The tool is worth the infrastructure cost (Qdrant, Voyage API, indexing time) because it reduces agent context consumption on discovery tasks.

**Action:** Integrate clew MCP tools as the preferred first-pass search for agent workflows. Fall back to grep for pattern matching (Category C) tasks. Document the distinction in agent instructions.

### What a "Conditional viability" result would mean

Clew helps on some discovery tasks but not all. The advantage is real but inconsistent -- possibly dependent on how well the indexed codebase's vocabulary matches the scenario's language, or on whether the relevant relationships are captured in the trace graph.

**Action:** Use clew selectively for tasks that match its strengths (semantic discovery, call chain tracing). Keep grep as the primary tool for most tasks. Invest in improving the weak areas before broader adoption.

### What a "Not viable" result would mean

Clew does not provide meaningful value over grep even for discovery tasks. Either the semantic search doesn't bridge vocabulary gaps effectively, or the compact responses still carry too much noise, or the relational insight from trace doesn't justify the extra infrastructure.

**Action:** Re-evaluate the architecture. Consider whether the problem is retrieval quality (embeddings, chunking) or presentation (response format, ranking). If retrieval quality is the issue, investigate alternative embedding models or chunking strategies. If presentation is the issue, it may be fixable with smaller changes.

### What to watch for in Category C

If clew somehow scores well on Category C tests, something is wrong with the scoring methodology. These are grep's home turf. Honest Category C results (grep wins or ties) validate that the evaluation framework is fair.
