# V2.2 Viability Evaluation Results

**Date:** 2026-02-17
**Verdict: ITERATE (Strong)**
**Clew Average:** 3.89/5.0
**Grep Average:** 4.32/5.0
**Win/Loss:** 4 clew / 4 grep (corrected from 3-5 after meta-evaluation; see Confidence Notes)

**Previous:** V2.1 scored 1.36/3.0 (NOT VIABLE). V2 scored 2.17/3.0 (CONDITIONAL). V1 scored 1.72/3.0 (WEAK). Note: V2.2 uses a completely different scoring methodology (1-5 scale, 5 weighted dimensions, blind evaluation), so scores are not directly comparable to V2.1 and earlier.

---

## Executive Summary

V2.2 redesigned both the evaluation methodology and the tool's CLI output. The evaluation now uses blind scoring with anonymized transcripts, scenario-based prompts (not identifier lookups), and a 5-dimension rubric on a 1-5 scale. The tool gained compact JSON output (`--json`), same-file call resolution in the relationship extractor, and was re-indexed with 231 trace relationships.

**Results:** Clew averages 3.89/5.0 across 8 tests, tying grep at 4 wins each (corrected after meta-evaluation identified an A1 scorer arithmetic error). Grep averages 4.32/5.0, driven by a 1.50-point completeness advantage (5.00 vs 3.50). Clew wins decisively on semantic discovery tasks where the user's terminology does not match the code's vocabulary (A3: +1.05, B1: +0.55). Grep wins decisively on exhaustive enumeration (A2: -1.70, B2: -1.75) and pattern matching (C1: -1.45).

The gap is 0.43 points on average, driven entirely by completeness. On the other four dimensions combined, the tools are nearly tied. Clew finds the right code faster; grep finds all the code.

**Verdict: ITERATE (Strong).** Clew meets the Iterate threshold (avg >= 2.50 AND wins >= 3/8) but falls short of Ship (avg >= 3.50 AND wins >= 5/8). Under the original plan's category-specific criteria, clew's 3/4 Category A wins with a 4.22 average would meet the Strong Viability threshold. The path to Ship is clear engineering work (entity resolution, exhaustive mode), not research.

## Methodology

### Blind Evaluation Design

V2.2 uses the most rigorous evaluation methodology to date, with 24 total agents (16 explorers + 8 scorers):

1. **16 exploration agents** (2 per test x 8 tests): For each test scenario, two independent agents explored the codebase -- one using only grep/glob/read tools, one using clew CLI tools plus read. Neither agent had access to the test plan, CLAUDE.md, or any prior knowledge of the codebase. Each agent received only a natural-language scenario description.

2. **Anonymization**: After exploration, transcripts were anonymized. Each test's two transcripts were randomly assigned "Alpha" and "Beta" labels. The mapping between Alpha/Beta and clew/grep was recorded in a separate `anonymization-map.json` and not shared with scorers.

3. **8 independent scoring agents** (1 per test): Each scorer received both anonymized transcripts for a single test and scored each on 5 dimensions against an absolute rubric. Scorers did not know which agent used which tool. They were instructed to score against the rubric, not to compare agents to each other.

### Test Scenarios

8 scenarios across 3 categories:

| Category | Tests | Design Intent |
|----------|-------|---------------|
| **A: Discovery** (4 tests) | A1-A4 | Vague starting points where the agent must discover the right code. Clew-advantaged: semantic search can bridge vocabulary gaps. |
| **B: Structural** (2 tests) | B1-B2 | Known concepts requiring structural understanding. Mixed advantage: clew trace helps for call graphs, grep helps for exhaustive model usage. |
| **C: Grep-Advantaged** (2 tests) | C1-C2 | Pattern matching and exhaustive enumeration. Grep expected to win. Included for evaluation honesty. |

Specific scenarios:

| Test | Scenario |
|------|----------|
| A1 | Investigate subscription renewal -> PrescriptionFill creation failure path |
| A2 | Find all code that needs to change to add a new checkout source type |
| A3 | Map treatment recommendation generation logic and decision factors |
| A4 | Explain how order type determination affects downstream processing |
| B1 | Impact analysis: changing the Shopify order processing function signature |
| B2 | Map relationships between Order, PrescriptionFill, and Prescription models |
| C1 | Find all Django URL patterns that involve ecomm-related views |
| C2 | Find all Stripe API calls in the codebase |

### Scoring Rubric

5 dimensions, weighted:

| Dimension | Weight | What It Measures |
|-----------|--------|------------------|
| Discovery Efficiency | 30% | How quickly the agent reaches actionable understanding |
| Context Precision | 25% | Signal-to-noise ratio in loaded context |
| Completeness | 20% | Coverage of key artifacts per test checklist |
| Relational Insight | 15% | Connections surfaced that the agent did not explicitly search for |
| Answer Confidence | 10% | Whether the final output is actionable without further investigation |

Each dimension scored 1-5. The weighted average produces a per-test score.

### Viability Thresholds

| Verdict | Criteria |
|---------|----------|
| **Ship** | Clew avg >= 3.50 AND wins >= 5/8 |
| **Iterate** | Clew avg >= 2.50 AND wins >= 3/8 |
| **Kill** | Below Iterate thresholds |

## Category Breakdown

### Discovery (A1-A4): Clew 4.22 vs Grep 4.26

Nearly tied on average, but clew won 3 of 4 tests (A1, A3, A4). The sole grep win (A2) is an exhaustive enumeration task where grep's completeness guarantee gives it a decisive advantage. On pure semantic discovery tasks — where the user's terminology does not match the code's naming — clew won every test.

- **Clew wins when terminology gaps exist.** A3 ("treatment recommendations") and A4 ("quiz submission handler" → `typeform_webhook_view`) demonstrate clew's core value proposition. Semantic search bridges vocabulary gaps that grep cannot.
- **Grep wins when the task is exhaustive enumeration.** A2 ("all code that needs to change for a new checkout source") is a fan-out task where grep's literal pattern matching guarantees completeness.
- **A1 corrected to clew win.** Clew found the core faster (5 vs 3 on discovery efficiency) but grep explored more periphery (5 vs 3 on completeness). The scorer miscalculated a tie (4.25 vs 4.25) and broke it for grep, but the actual weighted scores are clew 4.35 > grep 4.15. Corrected per meta-evaluation.

### Structural (B1-B2): Clew 3.75 vs Grep 4.35

Split 1-1 with grep having the higher average. The divide is sharp:

- **B1 (call graph tracing): Clew wins clearly.** Trace mapped the full call graph automatically. This is clew's strongest use case -- structural understanding of function-level call relationships.
- **B2 (model relationship mapping): Grep wins decisively.** Entity resolution failure (`PrescriptionFill` resolved to TypeScript generated type instead of Python model) and the approximate nature of semantic search (30/58 files vs 58/58) made clew fundamentally unsuited for this task.

### Grep-Advantaged (C1-C2): Clew 3.35 vs Grep 4.40

Grep won both, as expected. These tests validate the evaluation framework's honesty:

- **C1 (Django URL patterns):** Exhaustive import enumeration. Grep found 13 views; clew found 7.
- **C2 (Stripe API calls):** Negative proof. The correct answer is "zero Stripe calls exist." Grep confirmed this immediately via keyword search. Clew required 6 semantic searches that returned irrelevant results before falling back to grep.

Clew's one bright spot in Category C was relational insight on C2: trace on `log_payment_metrics` mapped the full Shopify -> NewRelic payment flow, providing architecture context grep did not surface.

## Fixes Applied Since V2.1

Three changes between V2.1 and V2.2:

### 1. Same-File Call Resolution (Python Extractor)

The relationship extractor now performs a pre-pass to collect all top-level function and class definitions within each file. When resolving call targets, it checks file-local definitions as a fallback after the import table. This fixes cases where `_helper_function()` calls within the same file were left unresolved or resolved to the wrong entity.

**Files changed:** `clew/indexer/extractors/python.py`

### 2. Compact CLI Output (`--json` flag)

Added `--json` flag to `clew search`, `clew trace`, and `clew status` commands. This produces compact JSON output with snippet-level results (signature + docstring preview) instead of the verbose raw JSON that caused V2.1's efficiency catastrophe. The `--full` flag can be combined with `--json` to include full content when needed.

**Files changed:** `clew/cli.py`

### 3. Re-Index with Relationship Fixes

The target codebase was re-indexed with the same-file resolution fix, producing 231 trace relationships (up from the V2.1 baseline). This improved trace quality for function-level queries.

## What Clew Does Well

### 1. Semantic Discovery Across Vocabulary Gaps

Clew's strongest signal is on tasks where the user's language does not match the codebase's naming conventions:

- A3: "treatment recommendation logic" -> `get_treatment_program_recommendations` (score 0.94, first result)
- A4: "quiz submission handler" -> `typeform_webhook_view` (score 0.84, first result, after grep's `quiz.*submit` regex failed entirely)
- B1: "function that processes shopify orders for treatment refills" -> semantic search returned the implementation, webhook entry point, and wrapper in a single query

In these cases, clew reached actionable understanding in 1-2 tool calls while grep needed 3-6 preliminary steps to find the right vocabulary.

### 2. Automatic Call Graph Mapping

`clew trace` eliminates manual breadcrumb-following for function-level call relationships:

- A3: Trace mapped 18 outbound dependencies and 3 inbound callers in 2 calls, replacing ~10 manual reads.
- B1: Trace automatically surfaced the complete call graph including all downstream handlers and async tasks.
- A1: Trace found the webhook -> processing -> fill creation chain that would have required multiple grep + read cycles.

### 3. Precision on Focused Searches

When clew works, it works with very low noise. Multiple tests showed clew returning 0 false positives where grep returned 7-30+ irrelevant matches (A2 model definition: 0 vs 7; A3 initial search: 1 result vs 72 file matches).

## What Clew Still Struggles With

### 1. Exhaustive Enumeration

The single largest weakness. When the task is "find ALL places that X," semantic search is approximate and grep is exact:

- B2: Clew surfaced 30/58 files touching PrescriptionFill. Grep found all 58 in one call.
- A2: Clew missed a production creation site (`heal_otc_treatment_only_orders` in `care/tasks.py`). Grep found all of them.
- C1: Clew found 7/13 views with ecomm dependencies. Grep found all 13.

Semantic search returns "most relevant" results, not "all" results. For impact analysis and audit tasks, this is a fundamental limitation.

### 2. Entity Resolution

When the same class name exists in both Python and TypeScript (e.g., `PrescriptionFill`), clew's trace resolves unpredictably:

- B2: `clew trace "PrescriptionFill"` resolved to the TypeScript generated type (`../../generated::PrescriptionFill`) instead of the Python Django model. This returned 1 inbound import from a React component instead of the 58-file Python ecosystem.

Entity resolution ambiguity is not just a trace problem -- it affects any query where the target name is shared across languages.

### 3. Negative Proofs

Proving "X does not exist in the codebase" requires exhaustive search. Semantic search cannot do this:

- C2: Clew's 6 semantic searches for Stripe-related code returned irrelevant results (Recharge API clients, subscription tests). Only after falling back to grep did the agent confirm zero Stripe calls exist.
- The scorer noted: "when the answer is 'this thing does not exist,' keyword search is inherently faster and more precise."

### 4. Completeness Depth

Even when clew wins on discovery speed, grep agents consistently explored more peripheral code:

- A1: Grep found retry mechanisms, SKU constants, Celery beat schedules, monitoring modules, and manual recovery commands -- all missed by clew.
- A4: Grep covered 6 Pydantic data models, constant mappings, 3 test files, and secondary survey endpoints that clew did not surface.

Grep's completeness average (5.00) vs clew's (3.50) is the largest dimension gap (-1.50). This reflects a structural trade-off: clew's ranked retrieval prioritizes relevance over exhaustiveness.

## Confidence Notes

### Methodological Strengths

1. **Intended blind scoring.** Transcripts were labeled Alpha/Beta with a separate anonymization map, but tool names and output formats were not redacted from transcript content. Scorers could identify which tool each agent used. Despite this procedural failure, examination of scorer reasoning by three independent meta-evaluators found no evidence that tool identity influenced scores in a systematic direction. Future evaluations should implement full tool-name redaction.

2. **Scenario-based prompts.** Agents received natural-language scenarios, not identifier lookups. This measures discovery value, not lookup speed -- a fairer test of clew's intended use case.

3. **Category C honesty check.** Grep won both grep-advantaged tests as expected. This validates that the scoring framework is not systematically biased toward clew.

4. **Consistent weights.** All 8 scorers used the same 5-dimension rubric with the same weights (30/25/20/15/10). Scores are computed mechanically from dimension ratings.

### Methodological Weaknesses

1. **Agents, not humans.** All 24 agents are Claude instances. They may share systematic biases in exploration strategy, tool preference, or summary quality. A grep agent that happens to guess the right keyword on the first try may outscore a clew agent that takes a methodical approach, or vice versa. Human evaluators would provide stronger signal.

2. **Same model family.** Both exploration and scoring agents use Claude. The scorers' assessment of "actionable understanding" and "relational insight" reflects one model's judgment, not a diverse panel.

3. **Clew agents had Bash permission constraints.** Several clew exploration agents encountered issues requiring retries when invoking `clew` CLI commands via Bash. This may have inflated clew's tool call counts in some tests, slightly disadvantaging clew on the Discovery Efficiency dimension.

4. **Clew agents had grep access.** The clew exploration agent had access to Bash (and therefore grep/glob) in addition to clew CLI tools. In several tests (B1, B2, C2), the clew agent fell back to grep. This means the comparison is "clew+grep vs grep-only," which is realistic (a clew user would still have grep available) but should be understood as evaluating whether clew augments a grep workflow, not whether clew replaces grep.

5. **A1 scorer arithmetic error (corrected).** The A1 scorer calculated "4.25 vs 4.25" in their reasoning, but the actual weighted scores from their dimensional ratings are clew 4.35 vs grep 4.15 — a 0.20-point clew lead, not a tie. The scorer then broke the nonexistent tie in grep's favor. Because the evaluation defines winners by weighted score, A1 is corrected to a clew win. This changes the overall record from 3-5 to 4-4.

6. **Grep gets relational insight scores.** The test plan specified that grep agents should score N/A on relational insight. In practice, scorers awarded grep agents relational insight scores (3-5) for manual code-reading that surfaced connections. This gives grep a dimension it was not designed to compete on. If grep's relational insight were zeroed out, the gap would narrow significantly.

7. **Small sample size.** 8 tests is enough to identify patterns but not enough for statistical significance. A single lucky/unlucky search result can swing a test's outcome.

### Meta-Evaluation

Three independent agents reviewed this evaluation for bias and accuracy:
- A **methodology critic** (rated evaluation quality 2.5/5.0) identified the anonymization failure and A1 arithmetic error
- A **pro-clew advocate** found the ITERATE verdict robust under alternative rubric weights but noted completeness scoring conflates exhaustive file coverage with adequate coverage of important code paths
- A **pro-grep advocate** argued the 30% discovery efficiency weight favors clew's designed strength while underweighting grep's completeness guarantee

A **synthesizer** resolved disputes: the anonymization failure is a credibility issue (not accuracy-distorting), the dimension weights are defensible as pre-registered, and the A1 correction is warranted. Under the original plan's category-specific criteria, clew's 3/4 Category A wins would meet Strong Viability. Under the document's more conservative global thresholds, the verdict remains ITERATE. Full meta-evaluation reports are in `/tmp/viability-eval/meta-*.md`.

## Iteration Targets

To reach Ship (avg >= 3.50 AND wins >= 5/8), clew needs to flip 1 more test from loss to win and maintain its average above 3.50. The closest candidates:

### 1. Fix Entity Resolution (impact: B2, potentially A2)

Disambiguate Python vs TypeScript entities. When both `backend/care/models.py::PrescriptionFill` and `../../generated::PrescriptionFill` exist, prefer the Python model for trace queries. Options:
- Language hint parameter: `clew trace "PrescriptionFill" --language python`
- Heuristic: prefer entities with more inbound/outbound edges
- Source priority: rank entities from source code over generated code

B2 is the most winnable flip. Clew's semantic search found the model, serializer, signals, and webhooks across 8 targeted queries. The trace failure was the critical gap. Fix entity resolution and B2 becomes competitive.

### 2. Add Exhaustive Mode (impact: A2, C1)

Add `--exhaustive` flag that combines semantic search with targeted grep for completeness guarantees. For "find all X" tasks, run semantic search first to understand what "X" looks like, then grep to ensure nothing is missed. This directly addresses the completeness gap (3.50 vs 5.00).

### 3. Reduce Trace Output Size (impact: B2)

B2's trace returned 172KB of relationships, which the scorer noted was "too large to be useful." Implement output truncation, pagination, or summary counts for large trace results. Consider a `--summary` flag that returns edge counts by relationship type instead of full edge lists.

### 4. Improve Negative Proof Handling (impact: C2)

When semantic search returns low-confidence results (all scores < 0.5), explicitly flag this in the output: "No high-confidence matches found. Consider using grep for exhaustive verification." This at least signals to the agent that semantic search cannot answer the question.

### 5. Surface More Peripheral Code (impact: A1, A4)

Clew consistently finds the core code but misses peripheral artifacts (tests, constants, admin, scripts). Options:
- After finding core results, automatically suggest related files via trace edges
- Include "related tests" and "related admin" in compact search output
- Increase default `limit` from 5 to 10 for discovery scenarios

**Caution on cross-version comparisons.** V2.2 changed three things at once:

1. **Scale:** 1-5 instead of 0-3. A V2.2 score of 3.89/5.0 (78%) is not directly comparable to V2.1's 1.36/3.0 (45%).
2. **Dimensions:** 5 weighted dimensions (discovery efficiency 30%, context precision 25%, completeness 20%, relational insight 15%, answer confidence 10%) instead of 4 equally-weighted dimensions (efficiency, accuracy, completeness, tool calls).
3. **Test design:** Scenario-based prompts where neither agent knows what to search for, vs identifier-based prompts where the baseline agent was given exact grep queries. This changes what is being measured: V2.2 measures discovery value; V1-V2.1 measured lookup speed.

The methodology change was necessary because V1-V2.1 tested the wrong thing (can clew do grep's job faster?) rather than the right thing (can clew help an agent discover code it doesn't know how to find?). But this means the score trajectory (1.72 -> 2.17 -> 1.36 -> 3.89) reflects methodology changes as much as product improvement.

What can be compared across versions:
- **Clew's structural weaknesses persist.** Entity resolution failed in V2.1 (tests 3.2, 3.3) and again in V2.2 (B2). Model-level trace remains broken.
- **Semantic search accuracy is consistently strong.** V2.1 had 5/5 top-1 hits. V2.2 had immediate correct results on A3, A4, B1.
- **Completeness remains the biggest gap.** V2.1's completeness (2.00/3.0) was the one dimension that held steady. V2.2's completeness (3.50/5.0 = 70%) tells the same story -- clew finds the right code but not all the code.
- **Efficiency improved dramatically** from V2.1 (0.21/3.0) due to compact JSON output fixing the response size catastrophe.
