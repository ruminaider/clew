# V4.2 Evaluation Handoff: Engine Fixes + Reformed Methodology

**Date:** 2026-02-21
**Evaluates:** V4.2 engine (reranker skip fix, ENUMERATION removal, expanded DEBUG)
**Methodology:** Reformed evaluation standards (behavioral metrics, stability analysis, counterfactual analysis, feature health cards)
**Goal:** Determine whether the engine fixes produce genuinely surgical augmentation and whether clew is viable under rigorous evaluation

---

## What Changed Since V4.1

V4.1 passed Ship (7/12 wins, 4.42 avg) but with suspect behavior: 49% escalation rate ("always grep"), ENUMERATION detection never activated (0/45), and the discovery rubric was relaxed between V4 and V4.1. V4.2 fixes the engine root causes and adds evaluation methodology reforms.

### Engine Changes (V4.1 → V4.2)

| Change | File | Impact |
|--------|------|--------|
| **Removed reranker variance skip** | `rerank.py` | Root cause fix. RRF outputs have variance 0.001-0.01 which always triggered the 0.1 threshold, skipping the Voyage reranker. Reranker now runs → `_compute_confidence()` uses calibrated 0-1 scores → escalation rate should drop from 49% to ~13%. |
| **Removed ENUMERATION auto-detection** | `intent.py` | Deleted `_is_enumeration()`, `ENUMERATION_PHRASES`, `_ALL_NOUN_RE`, `_NON_ENUM_ALL_PREFIXES` (~70 lines). 0/45 agent queries matched these heuristics in V4.1. ENUMERATION still available via explicit `mode="keyword"`. |
| **Expanded DEBUG intent detection** | `intent.py` | Added plurals to hard keywords (bugs, errors, crashes, exceptions). Added soft keywords (fail, issue, problem, investigate, diagnose). Added phrases ("not working", "doesn't work", "does not work"). Prevents false grep escalation on debugging queries (V4.1 E4 problem). |
| **Removed `low_variance_threshold`** | `models.py` | Cleaned up dead config field. |
| **Explicit exhaustive mode handling** | `engine.py` | `mode="exhaustive"` now has its own code path (always runs grep, not flagged as auto_escalated). Separated from post-hoc autonomous escalation. |

### Methodology Changes

| Change | File | Impact |
|--------|------|--------|
| **Behavioral metrics** | `VIABILITY-EVALUATION-STANDARDS.md` | Escalation rate, feature activation tracked as advisory diagnostics |
| **Stability analysis** | `VIABILITY-EVALUATION-STANDARDS.md` | P(noise) computation from historical flip rate; LOW CONFIDENCE flag |
| **Counterfactual analysis** | `scripts/counterfactual_analysis.py` | Grep marginal contribution measurement |
| **Feature health cards** | `VIABILITY-EVALUATION-STANDARDS.md` | Pre-registered health cards per feature, advisory only, caps at Iterate |
| **Rubric stability policy** | `VIABILITY-EVALUATION-STANDARDS.md` | Anchor freeze after Ship; rubric versioning |

## Scoring Rubric

### Dimensions & Weights

| Dimension | Weight |
|-----------|--------|
| Discovery Efficiency | 30% |
| Context Precision | 25% |
| Completeness | 20% |
| Relational Insight | 15% |
| Answer Confidence | 10% |

### Rating Scale

| Score | Discovery Efficiency | Context Precision | Completeness | Relational Insight | Answer Confidence |
|-------|---------------------|-------------------|--------------|-------------------|-------------------|
| 5 | ≤5 tool calls | ≤10% noise | All checklist artifacts found | ≥3 unexpected connections | Immediately actionable |
| 4 | 6-7 tool calls | 10-20% noise | ≥90% found | 2 connections | Needs 1 follow-up read |
| 3 | 8-12 tool calls | 20-40% noise | ≥70% found | 1 connection | Needs 2-3 follow-ups |
| 2 | 13-15 tool calls | 40-60% noise | 50-70% found | 0 but tool supports it | Needs significant follow-up |
| 1 | 15+ calls or gives up | 60%+ noise | <50% found | No relational capability | Not actionable |

### Winner Determination

```
weighted_score = (discovery * 0.30) + (precision * 0.25) + (completeness * 0.20)
              + (relational * 0.15) + (confidence * 0.10)

if clew_weighted > grep_weighted: winner = "clew"
elif grep_weighted > clew_weighted: winner = "grep"
else: winner = "higher completeness score"  # tiebreaker
```

## Test Scenarios

### Track A: Regression (8 tests)

**A1:** "A customer reported their subscription renewal didn't create a prescription fill. Investigate the code path from order webhook to PrescriptionFill creation and identify where failures can occur."

**A2:** "We need to add a new checkout source type for a partner integration. What code needs to change? Provide a comprehensive list of files and code locations that reference or depend on the checkout source concept."

**A3:** "The pharmacy API is returning errors for some refill orders. Where in the codebase do we call the pharmacy API, how do we handle errors from it, and what retry/monitoring logic exists?"

**A4:** "Explain how the order processing system decides what type of order it is (e.g., new subscription vs. refill vs. one-time purchase) and how that determination affects downstream processing."

**B1:** "We're considering changing the signature of the function that processes shopify orders for treatment refills. What would break? Find all callers, understand what arguments they pass, and identify any test coverage."

**B2:** "Map the relationship between Order, PrescriptionFill, and Prescription models. Include foreign keys, many-to-many relationships, through-tables, and any computed properties that bridge them."

**C1:** "Find all Django URL patterns that involve ecomm-related views. List each URL pattern with its view and URL path."

**C2:** "Find all Stripe API calls in the codebase. List each call site with the Stripe API method being called and the context in which it's used."

### Track B: Feature (4 tests)

**E1: Middleware Audit** (post-hoc escalation target)
> "Our application uses custom Django middleware for request processing. Map all custom middleware classes: what each one does, what order they run in, and how they interact with each other. Focus on application-specific middleware, not standard Django middleware. Also identify any middleware that conditionally skips processing based on request attributes."

**E2: Celery Task Inventory** (grep augmentation target — previously ENUMERATION)
> "Create a complete inventory of all Celery tasks in the codebase. For each task, identify: (1) the task function name and file location, (2) what arguments it accepts, (3) how it is enqueued (`.delay()`, `.apply_async()`, or `send_task()`), (4) what retry configuration it has (if any), and (5) the business purpose of the task."

**E3: Model Dependency Chain** (result enrichment target)
> "Starting from the Order model, trace how creating an order triggers the creation of related records. Map the full chain: which models are created as side effects of order processing, what service functions orchestrate this creation, and what happens if any step in the chain fails. Include the relationships between the created records."

**E4: Email Confirmation Debugging** (honesty check — grep should NOT trigger)
> "A user reported that their subscription order was processed successfully but they didn't receive a confirmation email. Investigate the email sending pipeline from order completion to email dispatch. Focus on identifying the specific failure path — where in the code could the email send fail silently? Do NOT catalog all email-related code in the codebase."

## Sanitization Rules

| Original | Sanitized |
|----------|-----------|
| `clew search "query" --json` | `search("query")` |
| `clew search "query" --project-root ... --json` | `search("query")` |
| `clew trace "entity" --json` | `search("code related to entity")` |
| `rg "pattern" --type py` | `search("pattern")` |
| `Grep(pattern="...")` | `search("...")` |
| `Glob(pattern="...")` | `find_files("...")` |
| `Read(file_path="...")` | `read("...")` |
| `"source": "grep"` in results | Remove field |
| `"source": "semantic"` in results | Remove field |
| `"context": "Called by: X \| Tests: Y"` | Normalize to `"related": "..."` (keep content) |
| Score fields (`"score": 0.94`) | Remove |
| `confidence`, `confidence_label` | Remove |
| `mode_used`, `auto_escalated` | Remove |
| `intent` field | Remove |
| `suggestion_type` | Remove |
| Qdrant/embedding metadata | Remove |

**Critical:** Add empty `"related": ""` to grep-tool results so the field's presence/absence doesn't identify the tool.

## Execution Pipeline

```
Phase 0: Pre-flight (verify environment, install code, check index)
  ├──→ Phase 1: Exploration (24 agents: 12 tests × 2)
  └──→ Ground-truth: Reuse V4/V4.1 checklists (target codebase unchanged)
Phase 2: Sanitization (12 agents)
  → Phase 3: Verification (1 agent, pass: ≤3/24 transcripts identified >60%)
    → Phase 4: Scoring (24+ agents: 2 scorers per test)
      → Phase 5: Computation (scripts/viability_compute.py)
        → Phase 5.5a: Feature health assessment (from raw transcripts)
        → Phase 5.5b: Counterfactual analysis (scripts/counterfactual_analysis.py)
          → Phase 6: Disagreement resolution (conditional tiebreakers)
```

### Phase 5: Computation Command

```bash
python3 scripts/viability_compute.py \
    .clew-eval/v4.2/scores/scores.json \
    .clew-eval/v4.2/mapping.json \
    --behavioral .clew-eval/v4.2/behavioral.json \
    --rubric R2 \
    --flip-rate 0.30
```

The `--flip-rate 0.30` uses the average of historical flip rates (V4→V4.1: 25%, V3→V4: 37.5%).

### Phase 5.5a: Feature Health Assessment

From raw (unsanitized) transcripts, collect:

1. **Escalation rate:** Count queries where clew output contains `"auto_escalated": true`. Compute `auto_escalated / total_eligible` (excluding LOCATION/DEBUG intent queries).
2. **Reranker activation:** Check if reranker ran (look for calibrated 0-1 scores in output, not tiny RRF scores 0.001-0.3).
3. **E4 DEBUG detection:** Verify E4 clew queries classify as DEBUG intent (grep should NOT trigger).

### Phase 5.5b: Counterfactual Analysis

For escalated queries, extract behavioral data and run:

```bash
python3 scripts/counterfactual_analysis.py .clew-eval/v4.2/behavioral_data.json
```

Input format for `behavioral_data.json`:
```json
{
    "version": "V4.2",
    "queries": [
        {
            "test_id": "A1",
            "query": "subscription renewal processing",
            "intent": "code",
            "auto_escalated": true,
            "results": [
                {"file_path": "ecomm/utils.py", "source": "semantic"},
                {"file_path": "ecomm/tasks.py", "source": "grep"}
            ]
        }
    ]
}
```

### V4.2-Specific Post-Hoc Questions

1. **Escalation rate:** What % of queries auto-escalated? Target: 10-20%.
2. **Reranker activation:** What % of queries had the reranker run? Target: ≥70%.
3. **E4 honesty:** Did DEBUG intent prevent grep escalation on E4 debugging queries?
4. **E2 without ENUMERATION:** Does E2 (Celery inventory) still get adequate coverage without ENUMERATION concurrent grep? (It may escalate via post-hoc grep if confidence is low.)
5. **Cross-version stability:** How many Track A tests flipped vs V4.1 and V2.3?
6. **Marginal contribution:** For escalated queries, what % of final results came exclusively from grep?

## Anti-Patterns to Avoid

1. **Do NOT** give exploration agents access to this handoff, test plans, CLAUDE.md, or any evaluation documentation
2. **Do NOT** run scoring before sanitization and verification complete
3. **Do NOT** let scorers compute aggregates or declare winners
4. **Do NOT** modify ground-truth checklists after exploration begins
5. **Do NOT** re-run individual tests that scored poorly (re-run ALL or NONE)
6. **Do NOT** make behavioral metrics verdict-affecting — they are advisory only
7. **Do NOT** compare V4.2 absolute scores to V2.3 scores (different rubric version R2 vs R1) — use win rates for cross-rubric comparison
