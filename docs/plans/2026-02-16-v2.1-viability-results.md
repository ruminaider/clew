# V2.1 Viability Test Results

**Date:** 2026-02-16
**Overall Score:** 1.36 / 3.0 (NOT VIABLE)
**Previous (V2):** 2.17 / 3.0 (CONDITIONAL)
**Previous (V1):** 1.72 / 3.0

## Executive Summary

V2.1 scores **1.36/3.0**, a regression from V2's 2.17. Search accuracy remains strong (2.00) with enrichment proving its value -- all 5 semantic search tests hit top-1. However, efficiency collapsed (0.21 vs 1.69) because clew's raw JSON responses are extremely verbose compared to baseline grep/read operations, and tool calls regressed (1.21 vs 2.81) because many baseline operations need only 1 grep call that clew can't beat. Structural trace remains the weakest area, with model/class-level resolution still broken (tests 3.2, 3.3, 4.2 all scored 0 across all dimensions).

**VERDICT: NOT VIABLE** -- overall average 1.36 < 2.0 threshold. High-value categories (2+6+7) average 1.21 < 1.5 threshold.

**Critical caveat:** This test used raw JSON responses, not compact mode. Compact mode could dramatically improve efficiency scores. See Confidence Notes.

## Scoring Methodology

### Efficiency Conversion
Baseline lines are converted to characters at ~80 chars/line for fair comparison with clew's character-based response sizes. Example: 52 lines = ~4,160 chars baseline. Clew's 33,834 chars for the same test = **8.1x worse**, not better.

For trace tests, "tool calls returning N relationships" are compared by information density: how much useful signal vs noise was returned.

### Tool Call Counting
Baseline tool calls are grep/read operations. Clew tool calls are `clew search` / `clew trace` / `clew explain` calls. Direct comparison: fewer clew calls = better.

## VERDICT: NOT VIABLE

**Score: 1.36/3.0** -- fails all thresholds.

### Why This Is Lower Than V2

The V2 test (2.17) was self-assessed by the same session that built V2, with a single subagent doing both baseline and clew testing. This V2.1 test used **independent validators with strict scoring**, which is more rigorous.

Key differences in methodology:
1. **Efficiency is scored against actual baseline operations**, not theoretical. Baseline grep/read operations return very little data (8-553 lines). Clew returns full JSON responses (7,700-181,579 chars). Even with compact mode, the raw search returns are verbose.
2. **Tool calls are counted mechanically.** Same count = 0% reduction = score 0. V2 may have been more generous.
3. **Structural failures are scored as 0**, not "partial". Tests 3.2, 3.3, 4.2 are complete failures.

### Root Cause Analysis

1. **Efficiency catastrophe (0.21/3.0):** Clew's raw JSON responses include full file content for each result. Even with `limit=5`, a single search returns 7K-180K chars. Baseline grep/read operations return exactly what's needed. The compact mode exists but was not tested -- these are raw responses.

2. **Tool call regression (1.21/3.0):** Baseline operations are highly targeted (grep for exact string). Many clew operations require the same number of calls (1 vs 1) for simple lookups, yielding 0% improvement. Compound tasks sometimes need MORE clew calls than baseline.

3. **Structural trace failures:** Model-level trace (3.2, 3.3, 4.2) remains broken. Class-level `contains` edges were added in V2.1 but the test shows they don't help with:
   - Finding ALL users of a class (3.2) -- this is fundamentally a grep job
   - Finding inheritance hierarchies (3.3) -- entity resolution fails on unfamiliar names
   - FK chain traversal (4.2) -- trace doesn't understand Django model relationships

4. **Accuracy held steady (2.00):** Top-1 hits for all semantic search tests (1.1-1.3, 5.1-5.2). Enrichment clearly helps -- test 1.3 had the highest score (1.24). But model/class trace accuracy is 0.

### What Would Need to Change

To reach CONDITIONAL (2.0+):
- **Compact mode by default** in test harness -- could reduce response sizes 5-20x, potentially moving efficiency from 0.21 to 1.5+
- **Model relationship extraction** -- Django FK/M2M detection in extractor pipeline
- **Inheritance tracking** -- tree-sitter class hierarchy extraction
- **Usage indexing** -- track where classes/models are imported and used (not just function calls)

To reach STRONG (2.5+):
- All of the above, plus
- **Response size optimization** -- return snippets not full content in raw mode
- **Hybrid search+grep** -- for pattern matching tasks, fall back to grep automatically

## Confidence Notes and Disagreements

1. **Efficiency scoring may be unfair.** The test used raw JSON responses, not compact mode. If compact mode reduces response size by 20x (as documented), efficiency scores would be dramatically different. This test measures the raw API, not the intended user experience.

2. **Tool call scoring is strict.** The rubric says ">= 50% fewer = 3". Same count = 0. This hurts clew on simple single-call tasks where grep and clew both need exactly 1 call.

3. **Test 4.2 tool call score is anomalous.** Scored 3 (75% fewer calls) but returned 0 results. Mechanically correct but misleading -- fewer calls is meaningless when the result is empty.

4. **V2 comparison is apples-to-oranges.** V2 was self-assessed with potentially generous scoring. V2.1 uses independent validators with strict rubric application. A fair comparison would re-score V2 with the same methodology.

5. **Search accuracy is strong.** 5/5 semantic search tests hit top-1. This is genuinely good and validates the enrichment + reembed fixes. The weakness is entirely in structural trace and efficiency.

6. **Enrichment value is confirmed.** Test 1.3 (description-based lookup) scored highest (1.24), proving LLM enrichment improves semantic search quality. This is the V2.1 win.

## Recommendations

1. **Re-test with compact mode.** The efficiency catastrophe may be an artifact of testing raw responses. Compact mode is the intended user experience.

2. **Don't invest more in trace for models.** Model/class relationship discovery is fundamentally better served by grep. Consider adding a `grep` tool to the MCP server that wraps ripgrep for pattern-matching tasks.

3. **Focus on response format.** The search quality (accuracy, completeness) is good. The presentation (efficiency) is bad. This is a formatting problem, not a retrieval problem.
