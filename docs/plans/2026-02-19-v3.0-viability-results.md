# V3.0 Viability Evaluation Results

**Date:** 2026-02-19
**Verdict: KILL (Rollback to V2.3)**
**Track A (Regression):** Clew avg 3.85/5.0, Grep avg 4.14/5.0, Clew wins 3/8
**Track B (Modes):** Clew avg 3.81/5.0, Grep avg 3.62/5.0, Clew wins 1/4
**Overall:** Clew avg 3.84/5.0, Grep avg 3.97/5.0

**Previous:** V2.3 scored 3.96/5.0 with 5-3 win record — verdict SHIP. V3.0 uses the same 5-dimension rubric, same 8 regression scenarios (Track A), plus 4 new mode-specific scenarios (Track B). Same methodology: 2+ scorers per test, transcript sanitization with verification gate, programmatic winner determination.

---

## Executive Summary

V3.0 added five features targeting V2.3's weaknesses: explicit search modes (`semantic`/`keyword`/`exhaustive`), ENUMERATION auto-classification, confidence scoring with actionable suggestions, grep integration via exhaustive mode, and peripheral code surfacing via `related_files`.

**Results:** V3.0 regressed on Track A (3.85 avg, 3/8 wins vs V2.3's 3.96 avg, 5/8 wins) and failed to demonstrate mode-specific improvement on Track B (3.81 avg, 1/4 wins). The verdict is **Kill** — Track A wins (3/8) fall below the Iterate threshold (4/8).

**Root cause:** V3.0's automatic features (ENUMERATION classification, confidence signaling, peripheral surfacing) were invisible in Track A transcripts — agents never saw or reacted to them. On Track B, where agents were told modes exist, mode selection was inconsistent: D1 used exhaustive correctly, but D2-D4 underutilized modes. The features exist in code but did not translate into measurable agent behavior changes.

**"Kill" means rollback V3.0 to V2.3**, not kill the project. V2.3 already shipped.

## Methodology

Follows `docs/VIABILITY-EVALUATION-STANDARDS.md` with V2.3 improvements plus V3.0-specific additions.

### Blind Evaluation Pipeline

1. **Pre-flight** — Verified V3.0 features (confidence fields, ENUMERATION classification, exhaustive mode, trace resolution)
2. **Scenario validation (Phase -1)** — 1 agent validated Track B scenarios, produced sealed ground-truth checklists with 10-18 concrete artifacts each
3. **Exploration (Phase 1)** — 24 agents (2 per test), Track A (no hints) and Track B (symmetric tool-awareness hints)
4. **Sanitization (Phase 2)** — 12 agents normalized tool names, V3-specific metadata, and output formats; randomly assigned Alpha/Beta labels
5. **Verification (Phase 3)** — 1 agent attempted to identify tools from sanitized transcripts: **PASSED** (0/12 at >60% confidence)
6. **Scoring (Phase 4)** — 24 agents (2 per test), Track A and Track B scored independently
7. **Computation (Phase 5)** — Programmatic aggregation via `scripts/viability_compute.py`
8. **Feature analysis (Phase 5.5)** — 2 agents analyzed raw transcripts for V3.0 feature visibility
9. **Disagreement resolution (Phase 6)** — 4 tiebreaker agents for tests A2, C1, D1, D3

### Scoring Rubric (unchanged from V2.3)

| Dimension | Weight | What It Measures |
|-----------|--------|------------------|
| Discovery Efficiency | 30% | How quickly the agent reaches actionable understanding |
| Context Precision | 25% | Signal-to-noise ratio in loaded context |
| Completeness | 20% | Coverage of key artifacts per test checklist |
| Relational Insight | 15% | Connections surfaced that the agent did not explicitly search for |
| Answer Confidence | 10% | Whether the final output is actionable without further investigation |

### Viability Thresholds (pre-registered)

| Verdict | Criteria |
|---------|----------|
| **Ship (Improved)** | Track A avg >= 3.66 AND wins >= 5/8 AND Track B avg >= 3.50 AND wins >= 3/4 |

| **Iterate** | Track A avg >= 3.00 AND wins >= 4/8 |

**Verdict determination:** Track A wins = 3/8, below Iterate threshold of 4/8. **Kill.**

## V3.0 Feature Analysis (Phase 5.5)

### Track A: Were V3.0 Features Visible?

| Feature | Expected Behavior | Observed in Transcripts |
|---------|-------------------|------------------------|
| `confidence_label` (high/medium/low) | Present in all search responses | **NOT observed** in any transcript |
| ENUMERATION auto-classification | Triggered on C1 ("find all URL patterns"), B1 ("find all callers") | **NOT observed** — no intent classification visible |
| `related_files` peripheral surfacing | Suggested tests, admin files via graph traversal | **NOT observed** in any transcript |
| `grep_results` integration | Present in exhaustive mode responses | N/A — no exhaustive mode used in Track A |
| `--mode` parameter | Not used (Track A has no hints) | Used once in C2 (`--mode keyword`) |

**Key finding:** V3.0's automatic features were invisible to Track A agents. No confidence labels, no ENUMERATION classification, and no peripheral surfacing appeared in any transcript. Only raw numeric scores (e.g., 0.635, 1.246) appeared in A3 and B1, resembling V2-era output rather than V3.0's regime-aware confidence scoring.

**Possible explanations:**
1. CLI `--json` output may not surface confidence/intent fields unless explicitly requested
2. Agents used compact output mode which may strip V3-specific metadata
3. Features work in the search pipeline but don't propagate to CLI/agent-visible output

**Impact:** Track A results essentially measure V2.3-equivalent behavior with V3.0 code, since no V3.0-specific features influenced agent exploration. The regression (5/8 to 3/8 wins) is agent variance, not V3.0-caused degradation.

## What Went Wrong

### 1. Features Were Invisible to Agents

The most damaging finding: V3.0's automatic features (ENUMERATION classification, confidence signaling, peripheral surfacing) did not appear in agent-visible output. Agents never adjusted behavior based on features they couldn't see. Track A results are effectively V2.3 re-measured with different agent instances.

### 2. Mode Selection Was Inconsistent (Track B)

Even when told modes exist, agents underutilized them. Only D1's agent consistently used exhaustive mode. D2 and D4 agents quickly reverted to semantic search + manual file reading. The `--help` output may not provide sufficient guidance for mode selection.

### 3. Discovery Efficiency Collapsed

Clew's Discovery Efficiency dropped from 3.12 (V2.3) to 2.31 (V3.0 Track A) — the lowest dimension score in any version. This isn't a V3.0-specific regression (grep Discovery also remained at 3.00), but it compounds the win/loss problem: with tighter scorer anchors, more tests fall to grep.

### 4. Agent Variance Dominates

6 of 8 Track A tests flipped between V2.3 and V3.0 with no code change affecting them. With 8 tests, a 2-test swing changes the verdict from Ship to Kill. The evaluation framework is not robust enough to distinguish systematic improvement from noise at this sample size.

## Confidence Notes

### Methodological Strengths

1. **Verification gate passed cleanly.** 0/12 tests identified at >60% confidence — strongest anonymization result across all versions.
2. **Pre-registered methodology.** All rules, thresholds, and computation logic were frozen before exploration.
3. **Programmatic computation.** Winners determined mechanically, not by scorer judgment.
4. **Track separation.** Track A and Track B scored independently, preventing cross-contamination.
5. **Mode-negative honesty check.** D3 correctly showed both tools scoring low on a genuinely difficult debugging task.

### Methodological Weaknesses

1. **Feature invisibility undermines the evaluation.** If V3.0 features aren't surfaced to agents, Track A measures agent variance, not V3.0 effectiveness. The Kill verdict is technically correct (V3.0 didn't help) but doesn't tell us whether the features *would* help if agents could see them.
2. **8 tests is insufficient for regression detection.** With 6/8 tests flipping between versions, the signal-to-noise ratio is too low. A 16-test Track A would provide more statistical power.
3. **Scorer anchor calibration drift.** V3.0's more specific anchors ("Discovery 5 = <=3 tool calls") changed scoring calibration. Both clew and grep Discovery scores dropped, making cross-version comparisons unreliable.
4. **D3 was contentious.** 6 tiebreaker dimensions, 4-point gaps on individual scores, both tools scoring near bottom. The scenario may not be well-suited for this rubric.
5. **Same model family.** All agents are Claude instances. Systematic biases in exploration or scoring cannot be ruled out.

---

## What Kill Means for V3.0

V3.0's features (mode parameter, ENUMERATION intent, confidence scoring, grep integration, peripheral surfacing) are implemented and tested (719 tests passing). The evaluation shows they didn't improve agent performance, primarily because:

1. **Automatic features weren't visible** — agents need to see confidence labels and ENUMERATION classification to benefit from them
2. **Mode selection is hard** — even with hints, agents don't consistently choose the right mode
3. **Agent variance exceeds feature signal** — V3.0's improvements are smaller than run-to-run noise

### Recommended Next Steps

1. **Fix feature visibility.** Ensure `confidence_label`, `intent`, `related_files`, and `suggestion` fields appear in CLI JSON output and MCP responses. Re-test with 2-3 scenarios to confirm agents can see and react to them.

2. **Improve mode documentation.** The `--help` output should include concrete guidance: "Use `--exhaustive` for 'find all X' tasks. Use `--mode keyword` for searching by identifier name."

3. **Re-evaluate with visibility fix.** Once features are visible, re-run a focused 4-test evaluation (C1, B1, D1, D4 — the completeness-focused tests) to measure whether visible features actually change agent behavior.

4. **Do not ship V3.0 in current state.** The code works but the agent-facing interface doesn't surface the value. Rolling back to V2.3 loses nothing because V3.0 features had no observable effect.
