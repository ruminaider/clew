# V3.0 Viability Test Plan: Mode-Aware Search with Grep Integration

**Date:** 2026-02-18
**Tests:** V3.0 code (confidence scoring, mode parameter, ENUMERATION intent, grep integration, peripheral surfacing)
**Goal:** Verify V3.0 maintains Ship quality on V2.3 scenarios AND demonstrates improvement on completeness-focused tasks via new search modes

---

## What Changed Since V2.3

V2.3 shipped at 3.96/5.0 (5/8 wins). Its weakest result was C1 (exhaustive URL enumeration: 3.13/5.0), exposing a fundamental gap: semantic search returns "most relevant," not "all matching." V3.0 adds features specifically targeting this gap:

| Change | Impact |
|--------|--------|
| **`mode` parameter** | Three explicit search modes: `semantic` (default), `keyword` (BM25-biased hybrid), `exhaustive` (semantic + ripgrep). Accessible via `--mode` flag or `--exhaustive` shortcut. |
| **ENUMERATION intent** | Queries like "find all X" or "list all Y" auto-classify as ENUMERATION, triggering 70/30 BM25/dense weighted hybrid with higher result limits. No explicit mode needed. |
| **Confidence scoring** | Search results include `confidence_label` (high/medium/low) and actionable suggestions ("Try mode='keyword' for broader matching" or "Try mode='exhaustive' for grep-based completeness"). |
| **Grep integration** | `mode="exhaustive"` runs semantic search first, generates grep patterns from query + results, then runs ripgrep. Returns `results` (semantic) + `grep_results` (additional grep-only hits) as separate keys. |
| **Peripheral surfacing** | Search results include `related_files` — tests, admin pages, and other connected files discovered via relationship graph traversal. |

**Expected impact by V2.3 weakness:**
- **C1 (exhaustive enumeration, 3.13/5.0):** `mode="exhaustive"` should dramatically improve completeness by combining semantic search with literal pattern matching.
- **B1 (caller enumeration, 3.55/5.0):** ENUMERATION auto-classification should find more callers; exhaustive mode provides completeness guarantee.
- **Discovery efficiency (3.12/5.0 avg):** Confidence signaling guides agents to try different strategies when initial search is low-confidence. Peripheral surfacing exposes related code without additional searches.

## Evaluation Structure

### Two Tracks

| Track | Tests | Purpose | Comparison |
|-------|-------|---------|------------|
| **A: Regression** | 8 (reuse V2.3 scenarios) | Verify V3.0 default mode is at least as good as V2.3 | V3.0 clew (default mode) vs grep |
| **B: Modes** | 4 (new scenarios) | Demonstrate V3.0 mode-specific improvement on diverse task types | V3.0 clew (mode-aware) vs grep |

**Why two tracks:**
- Track A prevents regressions. V3.0's automatic ENUMERATION classification and confidence signaling should help even in default mode, but the primary goal is "no worse than V2.3."
- Track B tests the V3.0 value proposition directly: do explicit search modes improve results across different task shapes? Track B includes exhaustive enumeration, keyword search, and a mode-negative honesty check.

### Combined Verdict

| Verdict | Criteria |
|---------|----------|
| **Ship (Improved)** | Track A: avg >= 3.66 AND wins >= 5/8 AND Track B: avg >= 3.50 AND wins >= 3/4 |
| **Ship (Maintained)** | Track A: avg >= 3.66 AND wins >= 5/8 AND Track B: avg >= 3.00 AND wins >= 2/4 |
| **Iterate** | Track A: avg >= 3.00 AND wins >= 4/8 (V3.0 slightly regressed but still usable) |
| **Kill** | Track A: avg < 3.00 OR wins < 4/8 (V3.0 regressed V2.3 — rollback needed) |

**Regression guardrail:** Track A Ship threshold is 3.66, not 3.50. This is V2.3's actual score (3.96) minus a 0.30 variance buffer. V2.3 demonstrated ±0.3 score swings between evaluation runs from agent variance alone. A Track A score below 3.66 indicates a real regression, not noise.

**Note:** "Kill" means "kill V3.0 and rollback to V2.3," not "kill the project." V2.3 already shipped.

## Methodology

Follows `docs/VIABILITY-EVALUATION-STANDARDS.md` with the same improvements established in V2.3:

- Pre-registered methodology (this document)
- 2+ scorers per test with tiebreaker protocol
- Transcript sanitization with verification gate
- Programmatic winner determination via committed script (no agent-computed aggregates)
- Relational insight scored for both tools

### Agent Models

All agents use the same model family for consistency and V2.3 comparability:

| Role | Model | Rationale |
|------|-------|-----------|
| Exploration agents | Sonnet 4.6 | Matches V2.3 exploration agents (Sonnet-class) |
| Sanitization agents | Sonnet 4.6 | Consistent with V2.3 |
| Verification agent | Sonnet 4.6 | Consistent with V2.3 |
| Scoring agents | Sonnet 4.6 | Consistent with V2.3 |
| Tiebreaker agents | Sonnet 4.6 | Consistent with V2.3 |
| Orchestration | Opus 4.6 | Coordinates pipeline execution |
| Scenario validation | Sonnet 4.6 | Pre-exploration codebase analysis |
| Post-hoc analysis | Sonnet 4.6 | Feature usage analysis |

### Methodology Refinements for V3.0

**1. Symmetric tool-awareness hints for both agents**

Both agents receive a tool-awareness hint to eliminate asymmetry in agent coaching:

Track B **clew** agent receives:
> "This tool supports multiple search modes optimized for different tasks. Run `clew search --help` to see available options including `--mode` and `--exhaustive` flags."

Track B **grep** agent receives:
> "Your search tools support various flags for different search strategies. Run `rg --help` to see available options including file type filters, context lines, and output modes."

Track A agents receive no tool-awareness hints (same as V2.3).

The clew agent must discover on its own which mode is appropriate for the scenario. This simulates a real user who knows modes exist but must choose based on the task.

**2. Sanitization of mode-specific artifacts**

V3.0 introduces new tool output that needs sanitization:
- `confidence_label`, `confidence`, `suggestion` fields → remove from sanitized transcripts
- `grep_results` key → normalize to generic `additional_results`
- `related_files` key → normalize to generic `suggested_files`
- `--mode exhaustive` / `--exhaustive` flag → normalize to `search("...", extended=true)`
- `--mode keyword` flag → normalize to `search("...", mode="broad")`
- `grep_patterns_used`, `grep_total`, `grep_capped` → remove

**3. Separate Track A and Track B transcripts during scoring**

Scorers receive tests from one track at a time. Track A and Track B use the same rubric but are scored independently. This prevents Track B's mode-specific behavior from influencing Track A scoring.

**4. Soft tool-call budget for exploration agents**

All exploration agents receive this instruction:

> "You have a budget of approximately 20 tool calls. Prioritize finding the most important artifacts first."

This is a soft cap — agents may slightly exceed it — but it standardizes exploration length across agents and makes Discovery Efficiency scoring more comparable. V2.3 had no budget, leading to some agents making 30+ calls while others stopped at 8.

**5. Scorer anchors (calibration note)**

V3.0 uses more specific scoring anchors than V2.3 (e.g., "Discovery 5 = ≤3 tool calls" instead of "Excellent — meets all criteria"). These reduce subjectivity but change scorer calibration. Cross-version score comparisons between V2.3 and V3.0 should focus on **win/loss patterns**, not absolute scores. The regression guardrail's 0.30 buffer accounts for potential calibration drift.

## Scoring Rubric

### Dimensions (unchanged from V2.3)

| Dimension | Weight | What It Measures |
|-----------|--------|------------------|
| Discovery Efficiency | 30% | How quickly the agent reaches actionable understanding |
| Context Precision | 25% | Signal-to-noise ratio in loaded context |
| Completeness | 20% | Coverage of key artifacts per test checklist |
| Relational Insight | 15% | Connections surfaced that the agent did not explicitly search for |
| Answer Confidence | 10% | Whether the final output is actionable without further investigation |

### Rating Scale

| Score | Discovery Efficiency | Context Precision | Completeness | Relational Insight | Answer Confidence |
|-------|---------------------|-------------------|--------------|-------------------|-------------------|
| 5 | Reaches understanding in ≤3 tool calls | ≤10% noise | All checklist artifacts found | ≥3 unexpected connections | Immediately actionable |
| 4 | 4-5 tool calls | 10-20% noise | ≥90% found | 2 connections | Needs 1 follow-up read |
| 3 | 6-8 tool calls | 20-40% noise | ≥70% found | 1 connection | Needs 2-3 follow-ups |
| 2 | 9-12 tool calls | 40-60% noise | 50-70% found | 0 but tool supports it | Needs significant follow-up |
| 1 | 12+ calls or gives up | 60%+ noise | <50% found | Tool has no relational capability | Not actionable |

### Completeness scoring: checklists and bonus discoveries

Track B tests use **ground-truth checklists** produced by the scenario validation agent (see Phase -1 below). Scorers compare agent findings against the frozen checklist.

If an agent discovers artifacts NOT on the frozen checklist, the scorer may award a **bonus** to the Completeness dimension (e.g., 4 → 5 if the agent found all checklist items PLUS additional valid artifacts). The checklist itself is not modified post-hoc.

Track A tests use the existing qualitative checklists from V2.3, which have been validated across multiple evaluation rounds.

### Winner Determination (unchanged from V2.3)

```
weighted_score = (discovery * 0.30) + (precision * 0.25) + (completeness * 0.20) + (relational * 0.15) + (confidence * 0.10)

if clew_weighted > grep_weighted:
    winner = "clew"
elif grep_weighted > clew_weighted:
    winner = "grep"
else:  # exact tie
    winner = "higher completeness score"  # tiebreaker
```

When 2 scorers disagree on winner direction, use the average of both scorers' weighted scores.

## Track B: Mode-Specific Scenarios

Four new scenarios testing diverse V3.0 capabilities: two exhaustive enumeration, one keyword identifier search, and one **mode-negative honesty check** where default semantic mode should outperform explicit modes.

Both agents receive symmetric tool-awareness hints (see Methodology section). The grep agent's hint is functionally cosmetic but ensures methodological symmetry.

**Scenario viability:** All Track B scenarios are validated against the target codebase before exploration begins (see Phase -1). Each scenario must have sufficient codebase content to be meaningful.

---

### Test D1: Celery Task Call Site Enumeration (exhaustive mode target)

**Scenario given to both agents:**

> "Find every place in the codebase where a Celery task is enqueued — called with `.delay()`, `.apply_async()`, or `send_task()`. For each call site, identify which task is being called, what arguments are passed, and the business context (what triggers this task to be enqueued). Provide a complete inventory."

**Qualitative checklist (ground-truth checklist produced by Phase -1 validation):**
- All `.delay()` call sites with task names
- All `.apply_async()` call sites with task names
- All `send_task()` call sites (if any)
- Arguments passed at each call site
- Business context for each enqueue (what code path triggers it)

**Why this tests V3.0 modes:**
This combines semantic understanding ("what business context triggers this task?") with exhaustive pattern matching ("find EVERY `.delay()` call"). Default semantic search will find some call sites but miss others where the task name doesn't match search terms. `mode="exhaustive"` runs semantic search first (to understand the task dispatch pattern), then grep (to find all `.delay()`, `.apply_async()` literals).

**Expected advantage:** Clew (exhaustive) > Grep > Clew (default)

---

### Test D2: Environment Variable and Settings Inventory (keyword mode target)

**Scenario given to both agents:**

> "Create a complete inventory of all custom environment variables and Django settings that control application behavior. For each setting, identify: (1) where it's defined (settings files, .env, os.environ defaults), (2) where it's read in application code (os.environ.get, django.conf.settings, etc.), (3) what behavior it controls. Focus on application-specific settings, not standard Django framework defaults."

**Qualitative checklist (ground-truth checklist produced by Phase -1 validation):**
- All custom settings defined in settings files
- All `os.environ.get()` and `os.environ[]` call sites
- All `settings.CUSTOM_SETTING` references in application code
- Mapping of setting name → defined location → read location → controlled behavior
- Settings with default fallback values vs required settings

**Why this tests V3.0 modes:**
Environment variables and settings are identifiers — exact strings like `ENABLE_FEATURE_X` or `settings.PHARMACY_API_URL`. BM25 identifier matching (keyword mode) should find them effectively because clew's tokenizer extracts code identifiers. Semantic search might miss settings whose names don't semantically match the query. `mode="keyword"` triggers ENUMERATION with higher limits and BM25-biased retrieval.

**Expected advantage:** Clew (keyword) >= Grep > Clew (default)

---

### Test D3: Email Confirmation Debugging (mode-NEGATIVE — default semantic should win)

**Scenario given to both agents:**

> "A user reported that their subscription order was processed successfully but no confirmation email was sent. Investigate the email sending pipeline to find where the failure might have occurred. Focus on understanding the specific failure path from order completion to email dispatch — do NOT exhaustively catalog all email-related code in the codebase."

**Qualitative checklist (ground-truth checklist produced by Phase -1 validation):**
- Order completion trigger point (where email send is initiated)
- Email sending function/service
- Template selection logic (which template for subscription confirmation)
- Error handling around email dispatch (what happens if send fails)
- At least one plausible failure mode (template missing, queue full, config error, etc.)

**Why this is a mode-negative test:**
This is a focused debugging task. The scenario explicitly says "do NOT exhaustively catalog all email-related code." An agent that uses `mode="exhaustive"` will receive grep results for every occurrence of "email" in the codebase — hundreds of irrelevant hits that waste the tool-call budget and degrade Context Precision. An agent that uses default semantic mode should trace the specific order→email path efficiently.

This is Track B's honesty check, analogous to Track A's C1-C2 (grep-advantaged). If clew wins D3 using exhaustive mode, the test design is suspect.

**Expected advantage:** Clew (default semantic) > Grep > Clew (exhaustive/keyword)

---

### Test D4: Authentication and Permission Mapping (exhaustive mode target)

**Scenario given to both agents:**

> "Find all API endpoints that require authentication and identify which authentication method each uses (token auth, session auth, OAuth, API key, permission classes, etc.). Include decorator-based auth (`@login_required`, `@permission_classes`), middleware-based auth, and any custom auth backends. Also identify any endpoints that are explicitly exempted from authentication."

**Qualitative checklist (ground-truth checklist produced by Phase -1 validation):**
- All `@login_required` decorations
- All `@permission_classes([...])` decorations
- DRF authentication classes (per-view and default)
- Custom authentication backends
- Middleware-based auth (session, token)
- Auth-exempted endpoints (`@csrf_exempt`, `AllowAny`, etc.)
- Mapping of endpoint → auth method

**Why this tests V3.0 modes:**
Authentication patterns span decorators, class attributes, middleware, and configuration. Semantic search understands "authentication" broadly but can't enumerate every decorated endpoint. `mode="exhaustive"` adds grep for `login_required`, `permission_classes`, `IsAuthenticated`, `AllowAny`, and other literal patterns.

**Expected advantage:** Clew (exhaustive) > Grep > Clew (default)

## Execution Pipeline

### Phase -1: Scenario Validation + Ground-Truth Checklists (1-2 agents, runs before exploration)

A validation agent analyzes the target codebase to verify each Track B scenario is viable and produce concrete checklists.

**For each Track B scenario (D1-D4):**
1. Run quick searches against the target codebase to confirm sufficient content exists (e.g., D1 needs ≥5 `.delay()` call sites, D2 needs ≥5 custom settings)
2. Produce a **ground-truth checklist** of specific named artifacts the agent should find (e.g., "task_send_refill_notification.delay() in orders/tasks.py line 45")
3. Record the checklist in a sealed file alongside the mapping file

**Viability threshold:** Each scenario must have ≥5 concrete artifacts in its ground-truth checklist. If a scenario fails validation:
- Replace with an alternative scenario from the same category (exhaustive/keyword/mode-negative)
- The replacement scenario must be validated before exploration begins
- Record the replacement and reason in the results document

**Output:** 4 ground-truth checklists (sealed before exploration) + viability confirmation per scenario.

**Timing:** Phase -1 runs in parallel with Track A exploration (see Parallel Execution Strategy below). Track B exploration does NOT start until Phase -1 completes.

### Phase 0: Pre-flight (5 min)

- [ ] Qdrant running: `docker compose up -d qdrant`
- [ ] V3.0 code installed: `pip3 install -e .`
- [ ] V3.0 features verified:
  - [ ] `clew search "test query" --json` returns dict with `confidence_label` and `confidence` fields
  - [ ] `clew search "find all models" --json` returns `intent: "enumeration"` (ENUMERATION auto-classification)
  - [ ] `clew search "test query" --exhaustive --json` returns both `results` and `grep_results` keys
  - [ ] `clew search "test query" --mode keyword --json` returns results with ENUMERATION intent
- [ ] Index current: `clew status` shows enriched index on target codebase
- [ ] Trace working: `clew trace "PrescriptionFill" --json` resolves to Python model
- [ ] Agent environment clean: test agents do NOT have access to this plan, CLAUDE.md, or any documentation

### Parallel Execution Strategy

```
Phase 0 (pre-flight)
  ├──→ Track A exploration (8 tests, starts immediately)
  └──→ Phase -1: scenario validation (Track B)
         └──→ Track B exploration (4 tests, starts when validation passes)
                  ↓ (both tracks complete)
Phase 2: sanitization (all 12 tests, parallel)
  → Phase 3: verification (1 agent, sequential)
    → Phase 4: scoring (all 12 tests, parallel)
      → Phase 5: computation (script, no agents)
        → Phase 5.5: post-hoc feature analysis (1-2 agents)
          → Phase 6: disagreement resolution (conditional)
```

Track A exploration and Track B validation start simultaneously. This shaves the validation time off the critical path.

### Phase 1: Exploration (24 agents, parallelizable per track)

For each of the 12 tests, launch 2 exploration agents:

**Grep agent (all tests):**
- Tools: Grep, Glob, Read (in target codebase)
- No Bash, no clew access
- Receives only the scenario description (Track A) or scenario + tool-awareness hint (Track B)
- Track B grep hint: "Your search tools support various flags for different search strategies. Run `rg --help` to see available options including file type filters, context lines, and output modes."
- No access to test plan, CLAUDE.md, or codebase documentation
- Working directory: target codebase root

**Clew agent (Track A: tests A1-C2):**
- Tools: Bash (for clew CLI), Read (for follow-up)
- Receives only the scenario description
- No access to test plan, CLAUDE.md, or codebase documentation
- No mention of modes, flags, or V3 features
- Working directory: target codebase root

**Clew agent (Track B: tests D1-D4):**
- Tools: Bash (for clew CLI), Read (for follow-up)
- Receives the scenario description PLUS the mode awareness hint:
  > "This tool supports multiple search modes optimized for different tasks. Run `clew search --help` to see available options including `--mode` and `--exhaustive` flags."
- No access to test plan, CLAUDE.md, or codebase documentation
- Working directory: target codebase root

**Soft tool-call budget:** All exploration agents receive: "You have a budget of approximately 20 tool calls. Prioritize finding the most important artifacts first."

Each agent produces a transcript (tool calls + reasoning + final answer).

### Phase 2: Sanitization (12 agents, 1 per test)

For each test, a dedicated sanitization agent:
1. Reads both raw transcripts
2. Randomly assigns Alpha/Beta labels (coin flip, recorded in mapping file)
3. Applies the normalization table
4. Produces two sanitized transcripts
5. Records Alpha/Beta → clew/grep mapping in sealed file

**V3.0 sanitization rules:**

| Original | Sanitized |
|----------|-----------|
| `clew search "query" --json` | `search("query")` |
| `clew search "query" --exhaustive --json` | `search("query", extended=true)` |
| `clew search "query" --mode keyword --json` | `search("query", mode="broad")` |
| `clew trace "entity" --json` | `search("code related to entity")` |
| `rg "pattern" --type py` | `search("pattern")` |
| `Grep(pattern="...")` | `search("...")` |
| `Glob(pattern="...")` | `find_files("...")` |
| `Read(file_path="...")` | `read("...")` |
| `confidence_label`, `confidence`, `suggestion` fields | Remove |
| `grep_results`, `grep_total`, `grep_patterns_used` keys | Normalize to `additional_results` |
| `related_files` key | Normalize to `suggested_files` |
| Score fields (`"score": 0.94`) | Remove |
| Qdrant/embedding metadata | Remove |

Output: 24 sanitized transcripts + 1 mapping file.

### Phase 3: Verification (1 agent)

A single verification agent reads all 24 sanitized transcripts and attempts to identify which tool each agent used. Reports confidence per test.

**Pass criteria:** Correct identification at >60% confidence on no more than 3 of 12 tests (25%).

**Escalation procedure if verification fails:**

1. **First failure (>3 tests identified):** Re-run Phase 2 with a stricter sanitization prompt. Specifically:
   - Normalize all query text to remove vocabulary differences (replace exact search terms with category labels)
   - Remove all intermediate reasoning that references tool behavior
   - Keep only: tool calls (normalized), tool results (content only), and final answer

2. **Second failure:** Proceed with a documented caveat, consistent with V2.3 precedent: "Anonymization incomplete. Behavioral fingerprints persist (query breadth, result structure, exploration strategy). Scoring rubric is absolute and tool-independent. V2.3 used the same proceed-with-caveat approach and found no evidence of scorer bias."

This escalation is pre-registered — no post-hoc justification needed.

**Note on Track B transcripts:** Track B clew agents may use `extended=true` or `mode="broad"` flags that have no grep equivalent. If the verifier identifies Track B tests primarily due to mode flags, this is acceptable — note in results but don't count toward the 3-test threshold.

### Phase 4: Scoring (24 agents, 2 per test)

For each test, 2 independent scoring agents:
1. Receive both sanitized transcripts (Alpha and Beta) for that test
2. Receive the scoring rubric with dimension-specific anchors
3. For Track B tests: also receive the frozen ground-truth checklist from Phase -1
4. Rate each transcript on 5 dimensions (integer 1-5)
5. Do NOT compute weighted averages or declare a winner

**Scorer prompt (exact text):**

> You are scoring two agent transcripts that attempted the same codebase exploration task. Score each agent independently on 5 dimensions using the rubric below. Provide integer ratings (1-5) for each dimension.
>
> **IMPORTANT:** Score each agent against the absolute rubric, not relative to the other agent. A score of 3 means "adequate" regardless of what the other agent scored. Do NOT compute aggregates or declare a winner — just rate each dimension.
>
> **Dimensions:**
>
> | Dimension | Weight | 5 (Excellent) | 4 (Good) | 3 (Adequate) | 2 (Weak) | 1 (Failed) |
> |-----------|--------|---------------|----------|---------------|----------|------------|
> | Discovery Efficiency (30%) | How quickly the agent reaches actionable understanding | Reaches understanding in ≤3 tool calls | 4-5 tool calls | 6-8 tool calls | 9-12 tool calls | 12+ calls or gives up |
> | Context Precision (25%) | Signal-to-noise ratio in loaded context | ≤10% noise | 10-20% noise | 20-40% noise | 40-60% noise | 60%+ noise |
> | Completeness (20%) | Coverage of key artifacts (see test checklist) | All checklist artifacts found | ≥90% found | ≥70% found | 50-70% found | <50% found |
> | Relational Insight (15%) | Connections surfaced beyond explicit searches | ≥3 unexpected connections | 2 connections | 1 connection | 0 but tool supports it | Tool has no relational capability |
> | Answer Confidence (10%) | Actionability of the final output | Immediately actionable | Needs 1 follow-up read | Needs 2-3 follow-ups | Needs significant follow-up | Not actionable |
>
> **Bonus discoveries:** If an agent finds valid artifacts not on the checklist, you may award a bonus to Completeness (e.g., 4 → 5) at your discretion. Do not penalize for missing items not on the checklist.
>
> For each transcript, output your 5 ratings as integers only. Example format:
> ```
> Agent Alpha: Discovery=4, Precision=3, Completeness=5, Relational=4, Confidence=4
> Agent Beta: Discovery=3, Precision=4, Completeness=4, Relational=2, Confidence=3
> ```

**Track separation:** Score Track A tests (A1-C2) and Track B tests (D1-D4) in separate scoring sessions. Do not mix tracks.

### Phase 5: Computation (programmatic, no agents)

The committed script (`scripts/viability_compute.py`) performs all aggregation:

1. Read all scorer ratings from structured input (JSON/CSV)
2. Average ratings from 2 scorers per dimension per test per agent
3. Compute weighted averages per test per agent
4. Determine winner per test (higher weighted average; completeness tiebreaker)
5. Unmask Alpha/Beta using sealed mapping file
6. Compute Track A and Track B averages separately
7. Compute overall averages
8. Apply verdict thresholds:
   - Ship (Improved): Track A avg >= 3.66 AND Track A wins >= 5/8 AND Track B avg >= 3.50 AND Track B wins >= 3/4
   - Ship (Maintained): Track A avg >= 3.66 AND Track A wins >= 5/8 AND Track B avg >= 3.00 AND Track B wins >= 2/4
   - Iterate: Track A avg >= 3.00 AND Track A wins >= 4/8
   - Kill: Below Iterate
9. Flag scorer disagreements (>1 point on any dimension)
10. Output: results table, per-test breakdown, verdict, disagreement report

### Phase 5.5: Post-Hoc Feature Analysis (1-2 agents)

After Phase 5 computation, a separate analysis agent reviews **raw (unsanitized)** Track A transcripts to measure V3.0 feature usage:

1. **Confidence signaling:** Did the clew agent receive `suggestion` fields? Did it act on them (e.g., switched to a different search strategy after a low-confidence result)?
2. **Peripheral surfacing:** Did `related_files` appear in results? Did the agent explore the suggested files?
3. **ENUMERATION auto-classification:** Which Track A queries auto-classified as ENUMERATION? Was the classification correct?

This analysis is qualitative and does NOT affect verdicts or scores. It's published alongside quantitative results to evaluate whether confidence signaling and peripheral surfacing are working as designed.

### Phase 6: Disagreement Resolution (conditional)

If any scorer pair disagrees by >1 point on any dimension:
1. Launch a third tiebreaker scorer for that test only
2. Use the median of 3 scores for each dimension
3. Re-run Phase 5 computation script

## Edge Cases

Pre-registered handling for known execution-time edge cases:

| Edge Case | Handling | Rationale |
|-----------|----------|-----------|
| **Clew agent uses grep via Bash** | Allowed. Sanitize `rg` commands as `search(...)`. | Same policy as V2.3. Realistic comparison: a clew user still has grep available. |
| **Tool timeout or rate limit** | Re-run the agent once. If it fails again, score the partial transcript as-is. | Infra failures shouldn't invalidate a test, but repeated failure indicates a real issue worth scoring. |
| **Agent gives up early** | Score as-is. A low-quality transcript is valid data. | An agent that can't make progress with the tool is scoring information, not a re-run trigger. |
| **Track B agent doesn't discover modes** | Score as-is. | The evaluation measures "does mode awareness help?" If the agent doesn't use modes, the answer is "mode awareness didn't help this agent." |

**General principle:** All exploration transcripts are scored as-is. No re-runs for suboptimal agent behavior (early termination, missed mode discovery, grep fallback). Transcripts are only re-run for infrastructure failure (Qdrant down, tool not installed, rate limit).
